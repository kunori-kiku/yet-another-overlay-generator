# Plan 3 — Phase 1b: node agent

Outline: [outline.md](outline.md) · Prerequisites: Plan 1 (signing) + Plan 2 (custody mode) merged ·
Branch stacked on Plan 2's.

## Goal

A minimal **node agent** that generates its own WireGuard keypair locally, registers only its public
key, pulls its own signed placeholder bundle, verifies the Ed25519 signature + SHA-256 before
applying, splices its local private key into `PRIVATEKEY_PLACEHOLDER`, runs the **existing
`install.sh` verbatim**, and reports result. Single-tenant, static config source (a directory or a
read-only HTTP endpoint), **no enrollment yet** (a configured node identity). Proves the
custody+signing chain end-to-end on a real host.

Reads from specs: controller/signing, controller/key-custody, artifacts/install-script

## Implementation steps

1. **NEW `cmd/agent/main.go`** (+ `internal/agent/` for logic; new deps quarantined here only).
   Subcommands: `keygen` (idempotent: generate WG keypair via `wgtypes.GeneratePrivateKey`, write
   private key 0600 to `/etc/wireguard/agent.key` if absent, print the public key),
   `pull` (fetch this node's bundle + `bundle.sig` from the configured source), and `apply`
   (verify → splice → run install.sh → report).
2. **Local keygen, public-key-only registration.** The private key NEVER leaves the host. "Register"
   in Phase 1b = write the public key to a configured location / POST to the static source (real
   authenticated enrollment is Plan 4).
3. **Verify before apply.** Recompute the canonical digest over the pulled bundle files, verify
   `bundle.sig` with the pinned tenant/instance public key via `bundlesig.Verify`, AND check
   `checksums.sha256`. Refuse to apply on any failure (fail-closed). Anti-rollback stub: refuse a
   bundle older than the last applied generation.
4. **Splice + apply.** Replace `PRIVATEKEY_PLACEHOLDER` in the node's own `[Interface]` with the
   locally-held private key (in-memory, written to the 0600 conf install.sh expects), then run the
   **existing `install.sh`** unchanged. Do NOT reimplement WG/Babel orchestration.
5. **Report.** Write/POST applied generation + result + health back to the source.
6. **Degradation:** on source-unreachable or verify-fail, keep last-good config (never tear down a
   running tunnel); fail-closed only on *new* applies.
7. **Spec** `docs/spec/controller/agent.md`: the agent lifecycle (keygen→pull→verify→splice→apply→
   report), fail-closed/keep-last-good contract, and the "thin wrapper over install.sh" boundary.

## Tests produced by this plan

- `internal/agent/verify_test.go`
  - **Lifetime:** perpetual (`internal/agent`).
  - **Guards:** apply is refused on bad signature / bad checksum / rollback; accepted on valid; the
    private key is never written outside the 0600 path.
  - **Retirement:** never (pins verify-before-root-apply).
- `internal/agent/splice_test.go`
  - **Lifetime:** subject-scoped.
  - **Guards:** placeholder is replaced by the local private key and the resulting conf parses;
    non-placeholder (air-gap) bundles are passed through untouched.
- **Real-host smoke (manual, recorded in the PR):** two Linux nodes; node keygens, pulls its signed
  bundle, applies, the overlay comes up, `wg show` confirms the spliced key. This gate cannot be CI'd.

## Definition of done

- [ ] CI green; verify/ splice unit tests pass.
- [ ] Real-host smoke recorded in the PR description (pull→verify→apply→report on two nodes).
- [ ] Agent is a thin install.sh wrapper (no reconciler); private key never transmitted.

## Out of scope

Enrollment, mTLS, tokens, Postgres, long-poll, frontend (all Plan 4); multi-tenant, KMS, stage/
promote, hardware-signed membership (Plan 5). Phase 1b uses a static, single-tenant config source.
