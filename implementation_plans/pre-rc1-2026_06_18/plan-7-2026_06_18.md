# plan-7 — Backend shrink + deployment split  (subject S1, milestone 1.7)

> Part of the pre-rc.1 program — see outline.md for mission, principles, and full sequencing.

## Prerequisites

- **plan-6 (1.6 local-mode-rewire)** MUST land first — it removes the four `fetch('/api/…')`
  calls at `frontend/src/stores/topologyStore.ts:680` (`/api/validate`), `:717` (`/api/compile`),
  `:781` (`/api/export`), `:827` (`/api/deploy-script`) and routes local-mode compute to the
  in-browser TS compiler. Deleting the routes before plan-6 bricks local mode (R1).
- **plan-4 (1.4 TS-compiler-reimplementation)** — the in-browser compiler that makes a
  backend-free local build *functional* (consumed by plan-6). Must be landed.
- **plan-5 (1.5 conformance-harness+CI)** — `internal/conformance/` must exist as a **required,
  green** CI gate before any route is deleted. It (plus `cmd/compiler`) keeps the Go pipeline
  build/test-exercised after the HTTP handlers are gone (R2). Per the spine, plan-5 lands before
  plan-4; both land before plan-6; plan-6 lands before this plan.
- This plan is the **last** step of the local-mode→browser migration track. Run it at the **tail
  of Subject 1's TS-migration track (after plan-6)**, and **before Subject 4's re-audit
  (plan-20/21/22)** so the audit sees the shrunken anonymous surface (the security delta is a
  Subject-4 input).
- **plan-8 (1.8)** is independent and may land in either order — this plan removes *anonymous
  reachability*; plan-8 fixes the *algorithms* (S2 caps, S3 cursor, B4 streaming), still reachable
  via operator + `cmd/compiler`.
- **This plan OWNS the air-gap route removal.** plan-6, plan-8, and plan-21 defer the deletion of
  `/api/validate|compile|export|deploy-script` (and `gateAirgap`/`compute`/`Server.operatorAuth`)
  to this plan; they do not delete those surfaces themselves.
- **Inputs assumed present (beta.8, DONE):** B1 top-level panic recovery on all muxes
  (`internal/api/server.go:185-200` `recovered`/`Handler`/`AgentHandler`) is in place; `/api/health`
  keeps its own per-route `wrap` for CORS independent of this plan.

## Goal

Make the **deployed** YAOG backend **controller-only** in its *reachable surface*. After the
TS-migration track (plans 1.4/1.5/1.6) has shipped, the frontend's LOCAL mode runs entirely in the
browser and never POSTs to the Go backend's air-gap compute routes. This plan deletes the four
anonymous air-gap compute routes (`/api/validate`, `/api/compile`, `/api/export`,
`/api/deploy-script`), their handler functions, and the three handler-local ZIP helpers from the
shipped server — so no anonymous/unauthenticated code path reaches the keygen/allocator/compiler
pipeline — while retaining `/api/health` (ungated liveness probe), `readTopology` (used by the
operator-only `HandleCompilePreview`), and `EnableStatic`/`YAOG_WEB_DIR` (serves the panel SPA). It
establishes a two-deployment story (a static SPA pinned to local mode via a `VITE_LOCAL_ONLY` build
flag, plus the controller server) and banks the security delta: S1/S2/S3/B4 are downgraded from
anonymous-remote to operator-or-CLI-trust on the deployed server (and absent entirely from the
backend-free static site). The algorithms themselves are **not** fixed here — that is plan-8.

## Implementation steps

> **COHERENCE NOTE — air-gap removal MECHANISM (owner-confirmable).** The source plan's prose was
> internally consistent on **single-profile DELETE** in its Decision 1 and disposition table, but
> the program-level framing elsewhere flirted with a `//go:build airgap` build-tag alternative.
> This plan **RESOLVES to a plain single-profile DELETE** of the four air-gap route *registrations*
> at `server.go:117-120` plus their handler functions and the three ZIP helpers — **no
> `//go:build airgap` complexity, no env flag, no conditional compilation in `server.go`**. Rationale:
> the build tag's only claimed benefit ("compute code physically absent from the binary") is false —
> the controller binary links the full pipeline via `controller.CompileSubgraph`
> (`compile.go:138-168`) regardless of any tag; a tag on the HTTP handlers buys nothing while
> imposing a permanent two-profile build/test matrix that pulls against plan-2 (structural
> simplicity). **`/api/health` stays registered and ungated** after the removal (`server.go:116`
> survives untouched, still wrapped by `wrap`). **FLAG: this single-profile-DELETE decision is
> owner-confirmable** — if the owner wants a build-tag profile instead, STOP and re-plan before
> Phase 1. Default to DELETE.

## Definition of done

1. The shipped server does **not** register or respond to `/api/validate`, `/api/compile`,
   `/api/export`, `/api/deploy-script` (404), verified by automated negative tests on
   `Server.Handler()`.
2. `/api/health` remains a public, ungated liveness probe on the controller, still CORS-wrapped (the
   `wrap` closure is preserved; `server.go:116` survives).
3. **No anonymous/unauthenticated route reaches the compile pipeline** in the shipped server, proven
   by the negative routing test. (The binary still links the pipeline via `CompileSubgraph`; only
   anonymous reachability is removed — no "code physically absent" claim.)

   handler-local ZIP helpers (`createExportZip`/`tarGzDirectory`/`makeSelfExtractingInstaller`) are
