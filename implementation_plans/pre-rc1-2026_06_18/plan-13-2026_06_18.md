## Prerequisites

- **Subject 1 (plans 1–9) and Subject 2 (plans 10–12) land first** per the program spine (Subject 3 follows
  Subject 2). This plan is the **FIRST plan of Subject 3** and the hard prerequisite for plans 14–18 (3.2–3.6),
  which add scenario specs onto this scaffold rather than re-solving bring-up.
- **plan-7 (1.7 backend-shrink + deployment-split) air-gap end-state must survive.** This plan's two-boot
  harness boots `cmd/e2eserver --mode airgap`, which **requires the air-gap compute routes
  (`/api/{validate,compile,export,deploy-script}`) to remain reachable with `s.operatorAuth==nil`**. If plan-7
  resolves its air-gap mechanism by *plain-DELETE* of those route registrations (rather than keeping them behind
  a build tag / runtime profile), there is **no air-gap boot for this plan to stand up** and the air-gap compile
  canary is impossible. This is a recorded cross-plan coherence issue (see Owner flags) — plan-7's file must be
  reconciled with this plan and plan-21 (4.2 DAST, which targets the `--mode airgap` oracle) before either
  builds. **Recommended end-state: keep the air-gap routes alive** (build-tag or runtime profile), since the
  DELETE's only stated justification (binary footprint) is contradicted by 3.1's and 4.2's hard dependence on an
  air-gap build.
- **No upstream code blocker on `main`.** Beta.8 already shipped B1 (fleet-mux panic recovery), F1 (`getTrustlist`
  keystone-sign-401 fix), C1 (babeld.conf byte-stability), and S4/S5/S6 (revoked-resurrection guard, enrollment-
  token purge-on-revoke, enrollment-token TTL cap). The rig **observes** that healthy stack and does not re-plan
  or re-test these. (If F1 were still open the controller-fleet canary's keystone path would flake — but it's done.)
- **Independent of the Subject-1 conformance harness** (plan-3 / `internal/conformance/`, Go↔TS byte-equality).
  That is a *different artifact* on a disjoint surface (two compiler implementations equal vs. the deployed product
  behaving correctly in a browser). They share no files; this plan must NOT touch conformance. They CAN proceed in
  parallel on isolated branches.

## Goal

Stand up the **first browser end-to-end (E2E) test layer the project has ever had**, plus the scripted full-stack
bring-up it runs against: a Go controller in a dedicated test mode, the built frontend served from that controller
(`EnableStatic`), and a real-or-mock `yaog-agent`. The deliverable is a deterministic, headless, CI-integrated
Playwright runner with fixtures and seed/teardown that any later Subject-3 plan (14–18 / 3.2–3.6) can extend by
**adding a `*.spec.ts` file** — not by re-solving bring-up. This plan builds the *scaffold only*: exactly two canary
specs, each exercising a real cross-stack path, prove the rig works; broad scenario coverage is explicitly out of
scope and lives in plans 14–18. It is deliberately separate from the TS-compiler conformance harness: both are "the
missing test infra," they cover disjoint surfaces, and this plan must not duplicate or block the conformance work.

## Definition of done

1. `cd frontend && npm run test:e2e` boots a hermetic full stack — a controller-ON `cmd/e2eserver` boot (operator
   + enrollment token + agent port + built panel) AND an air-gap-OFF boot (built panel + unauthenticated
   `/api/compile`) — and runs **two** canary specs green, headless, on a clean checkout.
2. Bring-up is fully scripted and deterministic — no manual steps, no `sleep`-based waits; readiness is gated on the
   per-boot `E2E_READY` line; teardown leaves no temp dirs or orphan processes.
3. A new `frontend-e2e` CI job exists in `.github/workflows/ci.yml`, runs headless, uploads report + traces on
   failure, and is green; required-immediately vs. advisory-until-determinism-gate recorded per the Phase 1 decision.
4. The Go test-mode seam is **test-only**: `cmd/e2eserver`/`cmd/e2eagent` are excluded from release builds
   (`release.yml:182-188` lists targets explicitly), marked TEST-ONLY, and reuse the real `internal/api` +
   `internal/controller` packages plus a SHARED operator-seed helper with `cmd/server` (extracted from
   `operator.go:74-80`, no copied bootstrap logic). No production handler/route/auth behavior changed.
5. The two-boot resolution is real and tested: the air-gap boot serves `/api/compile` UNauthenticated (200) and the
   controller boot serves it AUTH-gated (401 unauth), confirming the `EnableController`/`gateAirgap` split
   (`server.go:84,91-99`).
6. Specs drive mode/connection via the documented `controller-storage` seed contract (`mode`/`baseURL`/`agentBaseURL`,
