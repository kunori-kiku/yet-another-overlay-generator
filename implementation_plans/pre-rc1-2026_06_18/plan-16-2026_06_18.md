## Goal

Build the "pitfalls a human (or a flaky network) would actually hit" engine and run it
against the live full stack so the defects it surfaces are fixed *before* Subject 4's final
security audit re-runs. This plan delivers two coordinated test engines and the small
surgical fixes they justify: **Engine A**, a browser-side adversarial-usage Playwright
suite (network-fault injection — abort/delay/500/401/403/409/412 — concurrent operators,
races, stale-tab + back/forward nav, double-submit, and error-state rendering across the
controller panel's stateful flows: login, deploy, keystone signing, save-conflict,
enrollment, revoke); and **Engine B**, a pathological-topology generator + Go fuzz/corpus
that drives the compile path (`render.All` → `compiler.Compile` → renderers → export) with
malformed, oversized, degenerate, and adversarial inputs — both directly (Go) and through
`/api/compile` in local mode — to find inputs that hang, panic, or silently produce
wrong-but-valid output. The output is a **prioritized defect list with reproductions**
(`docs/spec/rc1/3.4-findings.md`) routed to the owning subjects (S1 DoS cap/context → plan
1.8; C2 re-enable-collision auto-heal → the Subject-1 compiler owner; everything else
tracked). This is a *hunt + fix* plan, not a feature plan: it MAY land small, surgical
robustness fixes (idempotency guards, abort wiring, context-cancellation seams, test-only
seams) and MUST file anything larger as a tracked finding for the owning subject.
