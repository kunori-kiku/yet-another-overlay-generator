## Design & approach

### The go/no-go decision: the explicit criteria

rc.1 is **GO** iff EVERY line below is satisfied; otherwise **NO-GO** (cut a beta.9 with the fix, or record an
owner-accepted-risk exception with written rationale). `RC1-GATE.md` encodes these as checkboxes the owner signs.

**A. Blocker-clean (zero confirmed blockers open).**
- **A1.** All six original rc.1 blockers (B1, S1, S4, S5, S6, F1, C1) are fixed **and have regression coverage**.
  B1 = the shipped top-level `recovered(mux)` wrap (`server.go:180-187`) returning `apierr.CodeInternalPanic`;
  the gate cites the EXISTING beta.8 protection and the controller-handler-panic regression test (a plan-8 /
  Subject-4 deliverable per `completeness.recommended_followups[0]`, **consumed** here — not authored by 4.3).
  S4/S5/S6 covered by the keystone/enrollment regression paths; F1 by the plan-15 (3.2) keystone-sign-after-refresh
  E2E case; C1 by the plan-5 conformance babeld.conf golden (`babel.go:118-134`). The gate lists the specific
  test/anchor pinning each.
- **A2.** **Zero NEW confirmed blockers** surfaced by Subject 1 (TS reimpl + the security delta — a TS cutover, if
  it lands, *removes* the anonymous S1/S2/S3/B4 air-gap surface per report §3b, recorded as a positive delta),
  Subject 2 (phone UX), Subject 3 (the full-stack pitfall hunt — the subject *designed* to find new blockers, incl.
  the plan-18 real-tunnel tier and plan-16's edge-case hunt), or Subject 4 (the re-audit, LAST). A "confirmed
  blocker" = a finding the plan-21 re-audit classifies at the bar the original six met (fleet-availability,
  fleet-trust, or controller-mode security/correctness). Each either lands a fix (→ beta.9, re-run gate) or gets a
  written owner-accepted-risk exception.

**B. Gates green AND required in CI.**
- **B1.** `ci.yml` job `go` green: gofmt-drift + `go vet` + `go test ./...`, **plus `go test -race ./...`** (the
  highest-value add; owned by THIS plan).

### The release-publish mechanism (the `--latest` owner override, 2026-06-18)

**Owner override (2026-06-18).** `RELEASING.md:54-56` defaults a pre-release (rc.N) to `--prerelease` and

The tag push triggers `release.yml`; its `release` job auto-creates the GitHub release via
`softprops/action-gh-release@v2` (`release.yml:245-254`). softprops, given neither `prerelease:` nor `make_latest:`,
creates a **non-prerelease** release, and GitHub then marks the newest non-prerelease tag as **"Latest"** — which
TODAY happens to demote beta.8 only by GitHub's "newest non-prerelease" heuristic (verified live: `gh release list`

holding Latest in place). Under the override we WANT rc.1 to be Latest, but we make it DETERMINISTIC rather than
heuristic. Two correct ways to assert rc.1 as Latest; the runbook uses **(a)** as primary, **(b)** as documented
fallback, and Task 4b adds the workflow-side belt:
- **(a) Pre-create as `--latest` BEFORE pushing the tag** (primary): `gh release create v2.0.0-rc.1 --latest
  --notes-file <notes>`, then `git push origin v2.0.0-rc.1`. softprops, finding a release already at that tag,
  **UPDATES** it (uploading assets) and preserves the Latest flag. rc.1 becomes Latest; beta.8 is demoted.
- **(b) Post-tag edit** (fallback): push the tag, let `release.yml` create the release, then immediately
  `gh release edit v2.0.0-rc.1 --latest` (promote rc.1, demoting beta.8).
- **(belt-and-suspenders, Task 4b):** the `make_latest:` expression on the softprops step so the workflow
  DETERMINISTICALLY promotes rc.1 (and any final `vX.Y.Z`) to Latest while keeping `-beta.`/`-preview.` tags off
  Latest, regardless of operator timing.

A bare `gh release create … --latest` issued AFTER the tag push is WRONG (collides with the already-created
release and ERRORs). The promotion of rc.1 to "Latest" over beta.8 is no longer an open owner question — it is the
locked 2026-06-18 owner decision, recorded as an explicit override of `RELEASING.md:54-56`.
