# plan-2 — Controller-mode Validate runs the in-browser validator

> Part of [outline.md](./outline.md). Design verified by workflow `w0s4u9zxq` (validate404), corrections folded in.

## Prerequisites
None (file-disjoint from plan-1). Branch off `main`.

## Goal
`topologyStore.validate()` POSTs `/api/validate` in controller mode, but that route is `//go:build airgap` — **404 in the shipped controller build** (e2e masked it because the e2e server is `-tags airgap`). Validation is purely structural (schema+semantic, no keys, no server), so run the **in-browser TS validator** (byte-pinned to Go by the conformance harness) in controller mode too.

2. `frontend/src/compiler/localEngine.ts:22-24` (the "controller mode UNTOUCHED" claim that this invalidates), `frontend/src/compiler/index.ts:389` (validate / localValidate).

## Implementation steps
1. `topologyStore.ts` validate(): run `localValidate(topo)` whenever `localEngineEnabled()` (both local AND controller mode); keep the `/api/validate` fetch block as the `VITE_YAOG_LOCAL_ENGINE='backend'` escape hatch only (preserve the Bearer+CSRF header attachment). Rewrite the inline comment.
2. Rewrite the two now-false docstrings: `topologyStore.ts:50-73` and `localEngine.ts:22-24` — validate() dispatches to the in-browser validator in controller mode too (key-free); compile/export/deploy keep their controller-mode refusal guards.
3. `topologyStore.local-engine.test.ts`: rewrite the 6.2 tests asserting controller-mode validate FETCHES `/api/validate` → assert **zero fetches** + a populated `valid` validateResult; ADD a test that the `backend` opt-out still fetches with Bearer+CSRF.
4. `frontend/e2e/controller-fleet.spec.ts` (or a small new `controller-validate.spec.ts`): a **client-side** guard — `page.route('**/api/validate', r => …fail/abort)`, then click Validate, assert a result renders. (Client-side because the airgap e2e controller boot registers the route + returns 200 when authed; only the `!airgap` Go test proves the shipped 404.)
