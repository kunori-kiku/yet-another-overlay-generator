# plan-5 — Per-node update-status surface (Fleet + Node detail) + opt-in poll

See `outline.md`. Milestone: observability surface. **Prerequisites:** plan-1 (`done`, the `in_rollout`
flag), plan-2 (`done`, the settings carry `targetAgentVersion` + canary set).

## Goal
Add a derived per-node **update-status chip** to the Fleet registry + a matching field on the node detail
page, plus an opt-in auto-poll so canary→applied transitions appear live. The chip is computed by a pure
`deriveUpdateState(node, settings)`; no new agent/wire field is added.

## Implementation steps

1. **Carry the new node fields** (`frontend/src/types/controller.ts` + the node mapping in
   `controllerClient.ts`): add `inRollout: boolean` and `targetAgentVersion: string` from plan-1's
   nodeJSON (safe defaults).

2. **`deriveUpdateState(node, settings)`** (pure, co-located, mirror `isDrifting`): returns one of
   `'off' | 'not-targeted' | 'pending' | 'applying' | 'applied' | 'failed' | 'stale'`:
   - `settings.targetAgentVersion === ''` → `off` (no chip / n-a — the safety contract).
   - not `node.inRollout` (prefer the server flag; fallback: `agentRolloutFleetWide ? true :
     agentCanaryNodeIds.includes(node.nodeId)`) → `not-targeted`.
   - `lastHealth` starts with `abandoned:` → `failed`.
   - `lastHealth` starts with `health-confirmed (probationary)` → `applying`.
   - version compare (strip leading `v`, ignore build metadata) `agentVersion === target` (or
     `lastHealth` starts with `self-updated to`) → `applied`.
   - in rollout, version `< target`, no failure marker → `pending`.
   - `stale` ONLY when last-seen is old AND not legitimately mid-update (do not flag a quiet
     re-exec'ing node as failed/stale — require both staleness AND absence of an `applying`/recent marker).

3. **Chip in `NodeRegistry`**: a new column + cell rendering the state (modeled on the badge cells),
   colored by severity (applied=green, applying/pending=amber, failed=red, stale=grey, off/not-targeted=muted).
   Tooltip with the raw reason. Add a matching `<dl>` field on `FleetNodeDetailPage`.

4. **Opt-in auto-poll** (decided): a 15–30s interval (place on `FleetPage` or the Shell) that calls the
   existing `refresh()`; **pause when `document.hidden`**; **clear on unmount AND on logout/mode-switch**
   (respect the session gate — never leak requests past logout). Gate it behind a small "Live" toggle
   (default on) so the operator can stop it.

5. **i18n:** `updateStatus.*` keys for each enum state + the Live toggle, in BOTH `en.ts` and `zh.ts`.

## Verification
- `cd frontend && npm run lint && npm run build` (bijection).
- **Manual smoke:** with a target set, a canary node shows `pending`→`applied` as its reported version
  flips; a node with an `abandoned:` health line shows `failed`; an out-of-rollout node shows
  `not-targeted`; empty target shows no chip; the Live poll updates without manual refresh and **stops on
  logout**.
