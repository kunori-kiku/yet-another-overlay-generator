## Owner mixed-mode smoke runbook (deliver in release notes)

Three nodes: two managed (agents) + one **manual** (no agent), within one controller topology.
1. Design: mark node C `deployment_mode: manual`, give it a pubkey + endpoint (or run the kit on C to
   generate them).
2. On C: `yaog-agent kit` → keygen → descriptor; register C's identity in the panel (or via the kit's
   direct register). Download C's signed bundle; the kit splices C's private key; `sudo bash install.sh`.
3. On A/B (managed): deploy as usual.
4. Assert:
   - A and B's WireGuard configs carry C as a `[Peer]` (pubkey + transit IP + allowed-IP); C's bundle
     carries A and B. Overlay ping A↔C and B↔C 0% loss; Babel routes converge.
   - The membership manifest includes C (`{NodeID, pubkey, bundleSHA256}`); managed nodes verify it.
   - The controller registry/served output never contains C's private key (zero-knowledge).
   - Panel shows C as "manual / unmonitored", NOT counted in "fleet converged"; if C's reporter is on,
     C's conditions appear.
   - Custody negative: the update-topology API still refuses any private key; clearing C's reporter
     leaves C with no controller credential.

## Definition of done

- Beta tagged, `release.yml` + `docker.yml` green, owner mixed-mode smoke passed, promoted to Latest,
  assets verified.
- Runbook delivered; STATUS + memory updated; both 2026_06_25 subjects archived to `_completed/`.

## Out of scope

- Any further mixed-mode ergonomics beyond Option C (e.g. controller-pushed manual bundles, an

## Hazard

This feature changed the enrollment/custody/keystone seam. Do NOT promote to Latest until the owner
smoke confirms (a) a manual node participates correctly AND (b) no managed-node regression AND (c) the
zero-knowledge + membership-signature invariants hold on real hosts.
