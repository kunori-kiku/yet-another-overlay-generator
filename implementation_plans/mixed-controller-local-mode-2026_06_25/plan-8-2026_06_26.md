# plan-8 — Self-update reliability: retry-without-restart + dual-source + stall-aware download

> **Folded-in rider (owner-directed, 2026-06-26).** While smoking beta.14 the owner reproduced "the
> former error in updating that requires me to restart the service": a post-apply agent self-update
> whose binary download from `gh-proxy.com` times out is deferred **once** and never retried until a
> manual `systemctl restart yaog-agent` (or a new Deploy). The owner asked to fold the fix into
> Subject 2 so the next beta — the one that ships mixed controller+local mode — actually self-rolls
> out. Thematically distinct from mixed-mode, but it **gates the rollout** of every future beta, so it
> ships in the **same release** (plan-7). Reviewed per-PR like every other plan.

## Root cause (confirmed in code + on the live fleet)

Live evidence (node `VM-HKG1...`, gen-35, beta.14):
```
agent: post-apply self-update deferred: download https://gh-proxy.com/...beta.14/yaog-agent-linux-amd64:
  context deadline exceeded (Client.Timeout or context cancellation while reading body)
agent: applied controller generation 35
```
The *config* applied; only the *binary self-update* failed, and it never recovered without a restart.

Three coupled defects + one latent:

1. **No retry without a new generation.** The deferred self-update runs in the apply path only —
   `agent.go:216` (step 7, `deferSelfUpdate = dec == updateAfterApply`) → `performSelfUpdate`. The
   daemon loop (`cmd/agent/main.go:605`) calls the cycle's apply path **only on a new generation**; a
   timed-out long-poll (stable generation) just re-polls (`main.go:621-624`). So a deferred download
   failure is attempted once and never retried until a new Deploy or a service restart. The
   `agent.go:217` comment "next cycle retries" is misleading — "next cycle" means next *applied*
   generation, not next poll.
2. **Single download source.** `performSelfUpdate` builds one URL: `ghProxy + ReleaseURL + asset`
   (`selfupdate.go:199`). If the proxy is slow/down there is **no direct-GitHub fallback**, even
   though GitHub is directly reachable from the fleet (the proxy exists only for proxy-dependent
   nodes).
3. **Hard 2-minute total download timeout.** `downloadTo` uses `http.Client{Timeout: 2*time.Minute}`
   (`selfupdate.go:38,425`). `http.Client.Timeout` is a **total** deadline covering the body read, so
   a ~10–15 MB binary over a slow proxy trips it mid-body — exactly the logged error.
4. **`State.SelfUpdateBlocked` does not clear without a new generation.** `recordSelfUpdateBlocked`
   (`agent.go:249`) writes the curated Blocked reason. A *new-generation* apply clears it implicitly
   (`recordSuccess` rebuilds a fresh `State` and carries forward only the self-update *custody* fields
   — floor, `PendingUpdate`, abandoned — not `SelfUpdateBlocked`), but on a **stable generation** —
   the same gap as #1 — nothing clears it, so the `selfupdate: Blocked` condition lingers until the
   next Deploy or a restart.

## Design (no shims; reuses the existing verified-fetch + decide + perform path)

### Part A — daemon-loop retry, decoupled from generations (fixes #1)

New `agent.RetryDeferredSelfUpdate(p *SelfUpdateParams, nodeID, stateDir string, verifiedFetch func() (map[string][]byte, error), stderr io.Writer) (attempted bool, err error)` in a new
`internal/agent/selfupdate_retry.go` (keeps `selfupdate.go` focused). It is the **same verified path
the apply uses**, minus the apply:

1. Guards: `p == nil` → no-op. `LoadState`; if `SelfUpdateBlocked == ""` → no-op (nothing armed → no
   wasted Fetch). If `PendingUpdate != nil` → no-op (a swap is in flight; the boot reconcile owns it).
2. `verifiedFetch()` → the cryptographically **verified** file map (`client.Fetch` + `VerifyBundle`,
   the SAME primitive as the startup health-gate `main.go:513-522`). A fetch/verify error → return it
   (keep last-good; retry next tick). **Custody: every retry re-fetches + re-verifies — no persisted
   or unverified pins ever drive a swap.**
3. `parseAgentCatalog(files)` + `decideSelfUpdate(cat, p.RunningVersion, floor, abandoned)`:
   - `updateSkip` / `updateRefuse` → no longer armed (already updated on a prior re-exec, or the
     target was abandoned/refused): **clear `SelfUpdateBlocked`** (fixes #4) and return.
   - `updateAfterApply` / `updateForced` → `performSelfUpdate(...)` (re-execs on success; on a pre-swap
     failure, re-record the curated blocked reason via the existing `recordSelfUpdateBlocked` so the
     latch refreshes and the next tick retries).

Daemon loop (`main.go`): after a cycle that **did not apply** and **did not error**, on a backoff
(`--selfupdate-retry-interval`, default 10m, ≤0 disables; mirrors `--telemetry-interval`), call
`RetryDeferredSelfUpdate`. **It runs on the MAIN loop thread, never a goroutine** — a swap is
`syscall.Exec` and must never interrupt a mid-flight `install.sh`. Gating on `!applied` avoids
double-attempting right after the apply path already tried (step 7). Errors are logged best-effort.

The forced (pre-apply) path already retries — a forced-update failure errors the whole cycle and the
daemon re-runs it on `errBackoff` (`main.go:616-619`) — so Part A targets the **deferred** path; the
forced path still benefits from Parts B and C.

### Part B — direct-GitHub fallback (fixes #2)

In `performSelfUpdate`, try an **ordered** candidate list and take the first that downloads:
`[ghProxy+ReleaseURL+asset` (only if `ghProxy != ""`)`, ReleaseURL+asset]`. Proxy first (respects the
operator's proxy intent for proxy-dependent nodes), direct GitHub as the fallback. The
**SHA-256-vs-signed-pin** check still gates the swap **after** the loop regardless of which source
served the bytes, so multi-source is custody-safe (a tampered mirror fails the pin). The wrapped error
keeps the `"download "` prefix so `classifySelfUpdateBlock`'s download case still matches.

### Part C — stall-aware (idle) download timeout (fixes #3)

Replace the hard total `http.Client.Timeout` with: a `http.Transport.ResponseHeaderTimeout` (bounds
time-to-first-byte — a hung mirror fails fast) + a `context`-bounded absolute ceiling (generous, e.g.
15m) + a **`stallReader`** watchdog around the body copy that cancels the request if **no bytes are
read for `selfUpdateStallTimeout`** (reset on each chunk). This tolerates a slow-but-progressing large
binary while still bounding a truly hung transfer — the correct fix for the "deadline exceeded while
reading body" failure. `stallReader` is a small, testable `io.Reader` wrapper (not a shim).

### Part D — clear the Blocked reason on the no-new-generation path (fixes #4)

`RetryDeferredSelfUpdate` clears `SelfUpdateBlocked` (via `clearSelfUpdateBlocked`) when the target is
no longer armed (skip/refuse) — this is the missing clear for the stable-generation case. The
new-generation case is **already** handled (`recordSuccess` rebuilds `State` without carrying
`SelfUpdateBlocked`), so no change to `recordSuccess` is needed — adding one would be redundant.

## Files

- `internal/agent/selfupdate.go` — Part B (candidate-URL loop in `performSelfUpdate`); Part C
  (`downloadTo` stall-aware rewrite + `stallReader` + new timeout constants; drop the 2-min total).
- `internal/agent/selfupdate_retry.go` (NEW) — Part A (`RetryDeferredSelfUpdate`) + Part D
  (`clearSelfUpdateBlocked` helper, called on skip/refuse).
- `cmd/agent/main.go` — Part A daemon wiring: a `verifiedFetch` closure (refactor `healthCheck` to
  reuse it, DRY), the `--selfupdate-retry-interval` flag + opts field, the backoff-gated main-thread
  retry call.
- `internal/agent/selfupdate_test.go` / `selfupdate_retry_test.go` (NEW) — tests below.
- `docs/spec/operations/` — document the retry cadence, dual-source fallback, the stall timeout, and
  the `--selfupdate-retry-interval` / env knob.

## Tests

- `downloadTo` (httptest): succeeds on a slow-but-progressing body; the **stall** watchdog fires on an
  idle body (returns a clear "stalled" error, not an opaque ctx-cancel); `ResponseHeaderTimeout` on a
  mirror that never sends headers; non-200 → error.
- `performSelfUpdate`: proxy URL fails → **falls back to direct** and swaps; a tampered fallback
  (wrong bytes) is still **refused by the pin**; `ghProxy==""` → single direct URL.
- `RetryDeferredSelfUpdate`: no-op when `SelfUpdateBlocked==""` / `PendingUpdate!=nil` / `p==nil`;
  **clears** Blocked on `updateSkip`; attempts `performSelfUpdate` on `updateAfterApply` (injected
  `execFn`); re-records Blocked on a download failure; surfaces a verify error without clearing.
- `recordSuccess` clears `SelfUpdateBlocked` (Part D).

## Custody / invariants preserved (PRINCIPLES — signed self-update)

- Every retry **re-fetches + re-verifies** (`VerifyBundle`) before any swap — pins always come from a
  freshly signed `artifacts.json`.
- The **SHA-256-vs-signed-pin** gate is unchanged and source-agnostic (Part B is safe).
- The crash-loop cap (`PendingUpdate.Attempts`), anti-downgrade **floor**, abandoned-target memory,
  and the in-flight guard are **untouched** — the retry funnels through the same `decideSelfUpdate` +
  `performSelfUpdate`.
- The swap stays on the **main loop thread** (never concurrent with an apply).
- No generation / checksum / token / telemetry / report semantics change.

## Out of scope

- The rollout arming model (still operator-armed via `release-pins`).
- Replacing the gh-proxy (we add a fallback, not a replacement).
- Self-update on non-amd64/arm64 arches (still fail-closed, `selfupdate.go:171`).
