# plan-2 — Manual-identity registration + custody gate + keystone membership

**Subject:** mixed-controller-local-mode-2026_06_25 · **Track:** Go · **Depends on:** plan-1.

## Goal

Give a manual node's identity (pubkey + endpoint) a controlled way INTO the controller without the
enroll-token ceremony, gate custody so a **public** key is allowed for a manual node while private
keys stay barred, extend the duplicate-pubkey invariant across manual + enrolled nodes, and include
manual nodes in the off-host signed membership manifest (owner D4).

## Implementation steps

### Step 1 — A manual-identity registration path

Add an operator-gated endpoint (NOT the node/agent mux) to register a manual node's identity —
`POST /api/manual-node` (operator-authed), body `{NodeID, WGPublicKey, Endpoint?}`:
- Reuses the registry write but WITHOUT the enroll-token burn (a manual node has no token ceremony).
- Sets a registry record the compiler treats as a manual-ready identity. Decide the cleanest model:
  either (a) reuse `NodeApproved` + a `Manual bool`/source marker on the registry `Node`
  (`store.go:67-91`), or (b) the compiler already reads the pubkey from the **topology** (plan-1
  Step 2), so registration may only need to record provenance + run the dedupe guard. Prefer the
  approach that keeps `enrolledSubgraph` admission (plan-1) the single source of "ready" and avoids a
  second parallel readiness path. Record the choice in the outline decisions log.
- Audit the registration (hash-chained log, like enroll `enrollment.go:233-238`).
- Operator-paste (panel, plan-6) and the on-box kit (plan-4) both call this endpoint.

### Step 2 — Dedupe across manual + enrolled

Extend `CheckWGKeyUnique` (`enrollment.go:256-281`) so a manual node's pubkey is checked against ALL
known pubkeys (enrolled + manual). A manual node must not be able to claim an enrolled node's key (or
vice-versa). Test both directions.

### Step 3 — Custody gate: allow public, bar private (manual)

- `internal/api/handler_deploy.go:54-59` — today refuses ANY private key (`CodeCustodyPrivateKey`).
  Keep barring private keys for ALL nodes (managed and manual). But a manual node legitimately carries
  a **public** key in the topology — ensure the update-topology path accepts a manual node's
  `WireGuardPublicKey` (it must survive into the design so plan-1 admission can read it) while still
  rejecting any `WireGuardPrivateKey`/`FixedPrivateKey`. Confirm the existing managed path still
  strips/forbids all key material; only the manual public key is the new allowance.
- Mirror in `frontend/src/lib/custody.ts` (`dropAllKeys`/`stripPrivateKeys`) in plan-6 — here, only
  the Go API gate.

### Step 4 — Keystone membership includes manual nodes (D4)

In `internal/controller/keystone.go:138-196` (`stageManifest`), include each manual node's
`{NodeID, WGPublicKey, BundleSHA256}` in the signed membership manifest. The `BundleSHA256` requires
the manual node's bundle to be rendered by the controller (plan-3 produces it) — order plan-3 before
the manifest digest can bind. If plan-3 isn't merged yet, gate the manifest inclusion behind the
bundle's existence and assert it once plan-3 lands. This preserves "the signature covers what runs":
a managed node verifying membership finds the manual peer attested.

### Step 5 — `apierr`/validator codes

Any new error code (e.g. `CodeManualNodeDuplicateKey`, `CodeManualNodeMissingPubkey` if not already
added in plan-1) goes in all three coordinated places (`apierr.go` const + `registry` map +
`allCodes` in `apierr_test.go`) so `TestRegistryBijection` passes. No orphan codes.

- Independent review (4-lens + security HIGH — this is the enrollment/custody/keystone seam) → fix →

## Out of scope

## Hazard

This plan touches the three security pillars (enrollment, custody, keystone). The dedupe extension
and the "public allowed / private barred" gate are the highest-risk edits — adversarially verify that
no path lets a private key in and no manual node can hijack an enrolled pubkey.
