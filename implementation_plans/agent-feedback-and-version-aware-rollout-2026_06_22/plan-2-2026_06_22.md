# plan-2 — Operator API exposure + panel generic conditions render

## Prerequisites

- **plan-1** MUST be done first. plan-1 defines `model.Condition{Type,Status,Reason,Message,Since}`
  (the closed shape + `classify()` curator, Decision D5), adds the `conditions` field to the agent
  report wire (`reportRequestWire` at `internal/agent/controller_client.go:91-99`) and the controller
  `HandleReport` ingest (`internal/api/handler_agent.go`), and stores `Node.Conditions` (+ a
  server-stamped `ObservedAt`) on `controller.Node` (`internal/controller/store.go:80-111`) persisted
  by both Store impls. **This plan reads `Node.Conditions` and re-serializes it on the operator nodes
  endpoint; it adds NO new condition producers.** If plan-1's field/struct names differ from what is
  written below (e.g. it names the server stamp `observed_at` vs `since`), this plan adopts plan-1's
  actual names verbatim — cite them at edit time.

## Goal

Surface the structured `Node.Conditions` channel on the operator-facing nodes endpoint and render it
generically in the panel. Add a `conditions []conditionJSON` array (snake_case wire) to the operator
`nodeJSON` view (`internal/api/wire_controller.go:94-118`) populated by `HandleNodes`
(`internal/api/handler_deploy.go:272-288`); mirror it as `NodeCondition` + `ControllerNode.conditions`
in the frontend types and the `mapNode` mapping. Build one reusable, **generic** `<NodeConditions>`
strip component (badge color driven by a `status→color` map, tooltip = the curated `message`, label =
`type` + `reason`) rendered in both the Fleet `NodeRegistry` row and the `FleetNodeDetailPage`. The
component is generic by construction — a `status→color` map plus an extension hook for richer
known-type chips — so plan-3 (selfupdate/wireguard) and plan-6 (mimic) need **no new rendering code**,
only condition producers. All changes additive + backward-compatible: old controllers omit the field,
old agents report no conditions, and a node with zero conditions renders nothing extra.

## Implementation steps

### A. Backend — emit `conditions` on the operator nodes endpoint

1. **Add `conditionJSON` + the `Conditions` field to `nodeJSON`** at
   `internal/api/wire_controller.go` (insert immediately after the `InRollout` field, `:117`, inside
   the struct; define `conditionJSON` just above `nodeJSON`, `:90`). The field is `omitempty` so an
   empty slice never appears on the wire (old-controller-shape compatible; matches the `agent_version`
   omitempty precedent at `:104`). Mirror plan-1's `model.Condition` field set + the server stamp it
   added (shown here as `observed_at`; adopt plan-1's actual name).

```go
// conditionJSON is the operator-facing view of one structured Node Condition (plan-1's
// model.Condition). It is the curated channel that replaces panel string-matching of the
// free-form last_health line: status drives the badge color, reason is the closed CamelCase
// code, message is the single length-capped human line (NEVER a raw stderr dump — see D5).
// observed_at is the controller's receive stamp (server truth, not agent-clock). All fields are
// plain data; no key material.
type conditionJSON struct {
	Type       string    `json:"type"`
	Status     string    `json:"status"`
	Reason     string    `json:"reason"`
	Message    string    `json:"message,omitempty"`
	Since      string    `json:"since,omitempty"`
	ObservedAt time.Time `json:"observed_at"`
}
```

	// a node that has reported none round-trips with the field absent, and the panel renders no
	// strip. Curated + length-capped upstream (handler_agent ingest); this view re-serializes only.

2. **Populate it in `HandleNodes`** at `internal/api/handler_deploy.go:274-287` — add a mapping inside
   the per-node loop. Build the `[]conditionJSON` from `n.Conditions` and assign it to the new field.
   If `n.Conditions` is nil/empty, leave the slice nil (omitempty drops it). Extract a small
   `func mapConditions(cs []controller.NodeCondition) []conditionJSON` helper in `wire_controller.go`
   (next to `conditionJSON`) so the mapping is one place and unit-testable:

```go
// mapConditions projects the stored controller.NodeCondition slice onto the operator wire view.
// nil/empty in ⇒ nil out (omitempty drops the field). It reads the embedded model.Condition fields
// plus the wrapper's server-stamped ObservedAt, copying verbatim — the curation/length-cap already
// happened at ingest (handler_agent), so this is pure projection, no re-classification.
func mapConditions(cs []controller.NodeCondition) []conditionJSON {
	if len(cs) == 0 {
		return nil
	}
	out := make([]conditionJSON, 0, len(cs))
	for _, c := range cs {
		out = append(out, conditionJSON{
			Type: string(c.Type), Status: string(c.Status), Reason: c.Reason,
			Message: c.Message, Since: c.Since, ObservedAt: c.ObservedAt,
		})
	}
	return out
}
```

```go

			Conditions: mapConditions(n.Conditions),
```

   Confirm `controller` is imported in `wire_controller.go` (check the import block; add
   `"<module>/internal/controller"` if absent — cite the real module path from `go.mod`).
   `controller.NodeCondition` embeds plan-1's `model.Condition` (Type/Status/Reason/Message/Since) and
   carries the server-stamped `ObservedAt time.Time` as the wrapper's only `time.Time` field;
   `mapConditions` reads the embedded fields plus that wrapper stamp. `Type`/`Status` are the

### C. Frontend — the generic `<NodeConditions>` component

6. **Create `frontend/src/components/deploy/NodeConditions.tsx`.** A pure, generic strip mirroring the
   `UpdateStatusChip` styling precedent (`UpdateStatusChip.tsx:11-19`). Color is driven by a
   `status→class` map with an explicit fallback for unknown statuses; the badge label is `type` +
   `reason`; the tooltip is the curated `message`. Export a pure `conditionVisual(status)` helper so
   the color logic is unit-testable in the `node` vitest env (no jsdom needed — the same reason
   `deriveUpdateState` lives in `lib/`). Keep the known-type extension hook explicit but unused here
   (plan-3/plan-6 fill it):

```ts
import type { NodeCondition } from '../../types/controller';

// STATUS_CLASS maps a condition status to a Tailwind badge class. The canonical set is
// ok/warn/error/unknown (the backend classify() is the source of truth); an unrecognized status
// falls through to the neutral 'unknown' look so a new status never renders blank. Mirrors
// UpdateStatusChip.CHIP_CLASS.
const STATUS_CLASS: Record<string, string> = {
  ok: 'bg-green-900/40 text-green-300 border-green-700',
  unknown: 'bg-blue-900/40 text-blue-300 border-blue-700',
  warn: 'bg-amber-900/40 text-amber-300 border-amber-700',
  error: 'bg-red-900/40 text-red-300 border-red-700',
};

// conditionVisual is the PURE status→class resolver (exported for the unit test; no DOM). Any
// unrecognized status ⇒ the neutral 'unknown' class so the strip is total over any backend status.
export function conditionVisual(status: string): string {
  return STATUS_CLASS[status] ?? STATUS_CLASS.unknown;
}

// conditionLabel is the badge text: "<type>: <reason>" (the closed code, not the message). Generic —
// no per-type branching, so a new condition type renders with zero code change. (plan-3/plan-6 may
// add a richer KNOWN_TYPE chip override via this same boundary — the extension hook.)
export function conditionLabel(c: NodeCondition): string {
  return c.reason ? `${c.type}: ${c.reason}` : c.type;
}

// NodeConditions renders a node's structured feedback as a wrapping badge strip. Empty ⇒ null (no
// "no conditions" filler in the table). The curated message is the tooltip (title); never a raw dump.
export function NodeConditions({ conditions }: { conditions: NodeCondition[] }) {
  if (!conditions || conditions.length === 0) return null;
  return (
    <span className="flex flex-wrap gap-1">
      {conditions.map((c) => (
        <span
          key={`${c.type}:${c.reason}`}
          title={c.message || undefined}
          className={`px-2 py-0.5 rounded text-xs border ${conditionVisual(c.status)}`}
        >
          {conditionLabel(c)}
        </span>
      ))}
    </span>
  );
}
```

   Design rationale (cite in the PR): generic by construction — `STATUS_CLASS` is the only color
   authority and `conditionLabel` has no per-type branch, so plan-3/plan-6 add **producers only**. The
   "richer known-type chip" extension hook is the documented `conditionLabel`/wrapper boundary, not a
   new render path — later plans override the chip for a known `type` here without touching callers.
