> re-review → CI green → merge.

## Goal

Expose `link_direction` in the panel exactly as the owner specified: clicking an edge shows a
direction select labeled with the REAL node names + arrows (`A ⇄ B` / `A → B` / `B → A`), the
canvas renders a visible directional style for single-linked edges (with a `data-testid` for e2e),
and the editor keeps users out of the validator's error states (hinting on `forward` without a
host).

> **AMENDED by D11 (owner, during plan-1):** there is NO stored `reverse` value. The select still
> offers all three choices, but picking `B → A` ("to(A)") performs an EXPLICIT EDGE FLIP: one
> store action that swaps `from_node_id`/`to_node_id`, mirrors the three pin pairs
> (`pinned_from_*` ⇄ `pinned_to_*` — allocation-stable: link identity and interface names are
> direction-agnostic), clears `compiled_port` + `endpoint_host`/`endpoint_port` (the old dial
> target is now the dialer), sets `link_direction: 'forward'`, and prefills the endpoint-host
> picker from the NEW dialed node's public endpoints. The flip helper is a pure, unit-tested
> function in `frontend/src/lib/` (sibling of `sanitizeLinkDirection`). The canvas arrow therefore
> ALWAYS equals the drawn = dial direction — no start-marker mechanics needed; `both` renders as
> today and single-linked edges get the direction chip + end marker only.

## Implementation steps

### 1. EdgeEditor select

Insert after the Endpoint Port field (`:246-281`), following the Transport select block verbatim in
structure:

- Options (values `''` / `'forward'` / `'reverse'`; `''` renders as the doubly-linked option):
  labels via `t(language, 'edgeEditor.linkDirectionBoth'|'…Forward'|'…Reverse', { from, to })`
  interpolating `selectedEdgeFrom?.name` / `selectedEdgeTarget?.name` with arrows
  (`{from} ⇄ {to}` / `{from} → {to}` / `{to} → {from}` — the arrow always points at the DIALED
  node).
- `onChange`: `updateEdge(selectedEdge.id, { link_direction: v === '' ? undefined : v })`.
  Selecting `reverse` ALSO clears `endpoint_host`/`endpoint_port` in the same `updateEdge` call
  (rule-5 avoidance) and shows a muted hint (`edgeEditor.linkDirectionReverseClearedHint`).
- When `forward` is selected and `endpoint_host` is empty, render the warning-styled hint
  (`text-[var(--warning)]`, pattern `:406-408`): the validator will require an endpoint host.
- A muted help line (`edgeEditor.linkDirectionHint`) explaining single-linking (who initiates; the
  suppressed side still routes).

### 2. Canvas directional style

- Thread `linkDirection` through the `flowEdges` memo (`TopologyCanvas.tsx:262-296`) into
  `CustomEdgeData` (`CustomEdge.tsx:23-46`).
- **Direction chip** in the label pill beside the role chip (`:151-201` pattern): `⇄` (both,
  muted/default), `→` / `←` (single-linked, accent) — with
  `data-testid="edge-direction-chip"` and the resolved direction as text content or
  `data-direction` attribute (e2e locates by testid, NEVER color/class — project lesson).
- **Markers:** `both` renders EXACTLY as today (D8 — zero cosmetic churn). Single-linked edges
  carry the arrow marker on the DIALED side only: `forward` keeps `markerEnd`; `reverse` swaps to
  `markerStart` using the existing inline defs (`:224-239`, `orient="auto-start-reverse"`), and
  drops `markerEnd`. Also neutralize the built-in `MarkerType.ArrowClosed` at
  `TopologyCanvas.tsx:292/:525` for the reverse case so the two marker mechanisms don't fight.

### 3. i18n

`edgeEditor.linkDirection`, `.linkDirectionBoth/Forward/Reverse`, `.linkDirectionHint`,
`.linkDirectionReverseClearedHint`, `.linkDirectionForwardNeedsHost` — in `en.ts` (canonical) +
`zh.ts` (parity). (`MessageKey` typing makes a missing key a build error.)

### 4. e2e

Extend the design spec: draw/select an edge → set each direction via the select → assert
(a) the direction chip appears with the right value (`getByTestId('edge-direction-chip')`),
(b) the persisted edge JSON carries `link_direction` (store or export),
(c) selecting `reverse` clears the endpoint fields,
(d) `forward` + empty host surfaces the validation error through the existing Validate flow
(`error.validation_edge_link_direction_forward_no_endpoint` text via `tValidationError`).
