# plan-3 — Behavioral proof (realtunnel c4) + docs (one PR)

> Part of [`outline.md`](outline.md). **Prerequisites:** plan-1 merged (the compiler suppression
> exists). Independent of plan-2. Branch `feat/link-direction-proof-docs` off `main`. One PR;
> workflow review → fix → re-review → CI green → merge.

## Goal

Prove the feature end-to-end on real kernel WireGuard: a netns scenario where BOTH nodes are
genuinely dialable (so the race is otherwise possible), the edge is `forward`-only, the suppressed
side's rendered conf carries NO `Endpoint =` line, and the tunnel still forms + routes from the
inbound handshake alone. Then document the field where users and future sessions will look:
`edge.md`, `peer-derivation.md`, and the bilingual wiki.

**Reads from specs:** compiler-allocation, model-validation (realtunnel harness itself is not
modeled in `specs/` — a known coverage gap; the harness files below are the ground truth).

## Implementation steps

### 1. Realtunnel scenario `c4`

- Fixture `test/realtunnel/testdata/c4-linkdir/topology.json`: two router nodes, BOTH with
  `has_public_ip` + `public_endpoints` (both dialable — without the feature the reverse peer WOULD
  dial back and could win), one enabled edge with `endpoint_host` set + `"link_direction":
  "forward"`.
- `TestC4LinkDirection` beside `TestC3OneDirectional` (`scenarios_test.go`), scenario key `"c4"`:
  - Assert `!reverseEndpointPresent(t, <to-node>, <from-node>)` — the suppressed side's conf has NO
    `Endpoint =` line for the from-node (and positively assert the from-node's conf DOES carry the
    edge endpoint).
  - `requireHandshakes` + `requireRouteConvergence` + `requireOverlayPing` — the tunnel forms from
    the inbound handshake alone and routes both ways.
- Add `TestC4LinkDirection` to ci.yml's **non-blocking additive tier** grep
  (`.github/workflows/ci.yml:200-207`, `continue-on-error` — same tier as C3; the required floor
  stays canary+negative, per outline D5/scope).
- Local run: `REALTUNNEL_SCENARIOS=c4 go test ./test/realtunnel/ -run TestC4LinkDirection -v`
  (needs the netns privileges the CI job provisions; if unavailable locally, CI is the gate —
  say so in the PR).

### 2. Docs

- `docs/spec/data-model/edge.md`: new `## Link direction` section after §TCP transport — the three
  values, `""`≡both, who dials whom (forward dials `endpoint_host`; reverse dials the from-node's
  `public_endpoints[0]`), the suppressed side keeps AllowedIPs/Babel, the six validation rules,
  allocation invariance (direction is allocation-blind), and the roaming rationale (why single-link
  kills the race). Add the `link_direction` row to §Field contract.
- `docs/spec/compiler/peer-derivation.md`: update the reverse-endpoint resolution + roaming
  operational note for the suppression gates (cite `peers.go` anchors).
- `docs/wiki.md` + `docs/wiki-zh.md`: a user-facing subsection — when to single-link (the
  accelerator/relay scenario), how the editor labels the choice, the validation errors you may see
  and what they mean. Both languages, same content.
