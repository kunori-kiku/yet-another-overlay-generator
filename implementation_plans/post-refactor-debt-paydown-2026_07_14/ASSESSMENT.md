#### [HIGH/security] Standalone install.sh reads artifacts.json (root .deb pin) without verifying it is in the signed checksums set — signature bypass to root package install
- **files**: internal/renderer/templates/install.sh.tmpl:368-419 (reads at 400-417); contrast internal/agent/verify.go:225-229 (the guard the shell lacks); internal/render/artifacts_json.go:60-66 ("" when no catalog); internal/artifacts/export.go:67-69 (artifacts.json only listed when non-empty)

Both only constrain files LISTED in checksums.sha256 — sha256sum -c never flags an extra, unlisted file, and the signature covers only the checksums content. The mimic fallback then reads $SCRIPT_DIR/artifacts.json (413-417) for the .deb release_url + asset + sha256 and installs it as root via apt-get (444), with NO check that artifacts.json is covered by checksums.sha256. #### [MED/robustness] Trust-list signature submission is not serialized under lockTenantOps, letting a concurrent re-stage desync the signed manifest from the staged bundles and silently promote a fleet-stranding (M_old, B_new) pair
- **files**: internal/api/handler_keystone.go:297-367 (HandleTrustListSignature); internal/controller/compile_stage.go:79 (stage holds lockTenantOps); internal/controller/compile_promote.go:62-68 (promote gate checks signature-vs-pin only); internal/controller/storecore.go:783-790 (PutSignedTrustList overwrites with no CAS)
- **evidence**: CompileAndStage, PromoteStaged, Enroll and Rekey all take `defer lockTenantOps(t)()` (grep confirms these four are the ONLY holders). HandleTrustListSignature does NOT — its wrappers `secure`/`op` (adapter.go:43) are auth+method only, no tenant lock. The handler does two separate kv-locked calls: stagedManifest read (reads collStagedTL) then PutSignedTrustList write (overwrites collStagedTL, storecore.go:788 is a blind saveJSON). Between them a concurrent operator CompileAndStage (holding lockTenantOps, but which does NOT block this handler) can commit new staged bundles B2 at gen G+1 plus a new staged manifest M2 via stageManifest. The signature handler, having read M1 and verified submitted==M1 and sig-over-M1, then writes (M1, sig1) back over M2. The staged slot is now (M1,sig1) while the staged bundles are B2. PromoteStaged (compile_promote.go:62-68) verifies only that (M1,sig1) verifies against the pin — it explicitly does NOT re-derive the staged bundles' digests vs the manifest (documented at compile_promote.go:26-31 'Do not mistake this controller gate for the trust root'). So the promote SUCCEEDS silently and copies (M1,sig1) to the served slot alongside the B2 bundles.
- **impact**: Fail-closed fleet stranding, recoverable: nodes fetch /config and get a B2 bundle paired with the M1 trust-list; VerifyMembership computes hex(sha256(B2 checksums)) and compares to member.BundleSHA256 in M1 (which binds B1's digest) -> mismatch -> every node refuses and keeps last-good (verify.go:397). No private-key leak, no signature forgery, no trust bypass (the agent's exact digest binding guarantees fail-closed, never accept-wrong). Requires a narrow self-race: an operator re-staging while a signature POST is mid-flight, followed by a promote on the stale signature; the operator sees NO error at promote, so the stranding is silent until nodes report apply failures. This is the exact class of 'interleaved stage/promote strands the fleet' the project already defended elsewhere with lockTenantOps (compile_stage.go:73-79 comment), left as a consistency gap on the sign path.
- **fix**: Wrap HandleTrustListSignature's read-verify-write in `defer lockTenantOps(tenant)()` (it is a controller-domain mutation like stage/promote), OR make PutSignedTrustList a compare-and-swap that refuses when the current staged canonical bytes/epoch differ from the ones just verified (re-read collStagedTL under the same lock and assert byte-equality before overwriting). Serializing the sign path with lockTenantOps is the smallest change and matches the existing custody-mutation discipline.
