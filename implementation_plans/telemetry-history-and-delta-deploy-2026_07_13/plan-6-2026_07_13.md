# plan-6 — Force redeploy (per-node + fleet) + pre-deploy preview dialog

**Goal:** the escape hatch and the confidence view around plan-5's skip.

## Changes
- **Force:** `HandleStage` gains a force parameter — `force=all` (fleet) and `force=<nodeID,…>`
  (per-node): forced nodes stage at a new generation even when their digest is unchanged (the
  existing single-shot agent rerun stays as the last-resort path; this makes force a first-class
  controller action). Audit records forced stages distinctly.
- **Pre-deploy preview:** an operator-gated dry-run endpoint (extend `HandleCompilePreview` or a
  sibling `HandleDeployPreview`): compiles, computes per-node digests, compares against current
  staged/served — returns `{topology_version, nodes: [{id, name, changed}]}` WITHOUT staging
  anything. The response also carries the plan-5 keystone state (`keystone_full_restage: true` when
  rotation/first-pin pends) so the dialog says "keystone rotation pending — a full redeploy will be
  staged" instead of "0 changed" (the review-blocker UX face). Bound to `topology_version` so the Deploy call can assert it deploys what was previewed
  (topology changed in between → the panel re-previews; the deploy itself never blocks on it).
- **Panel (Deploy dialog):** on open → preview → list "will update (N)" / "unchanged (M)" with node
  names + a per-node force checkbox + a fleet "Force all" toggle; the result toast shows the
  StageResult counts (plan-5). Node-detail: a "Force redeploy this node" action beside the existing
  per-node actions (clear-rekey precedent). i18n en+zh; `data-testid` locators.

## Tests
- Go: force staging (unchanged digest + force → new generation); preview endpoint (no state
  mutation — staged/served untouched byte-for-byte; digest compare correct; topology_version echo);
  auth gates.
- FE: vitest for the preview→dialog state mapping (pure logic in `src/lib/`); Playwright e2e —
  deploy dialog shows changed/unchanged split (mock API), force toggle wires the param.
