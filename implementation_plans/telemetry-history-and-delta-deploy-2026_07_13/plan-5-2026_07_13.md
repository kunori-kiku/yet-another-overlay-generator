# plan-5 — staging skip-unchanged (per-node digest compare; keystone-seam proofs)

**Goal:** Deploy re-stages ONLY nodes whose bundle content changed; unchanged nodes keep their
current generation so agents see nothing new and never refresh. Agent (`cycle.go`) UNTOUCHED.

## Design (assessment recap)
Content identity = `hex(sha256(checksums.sha256))` — already the off-host-bound identity
(`compile.go:24-32`); `compiled_at` is already outside the checksummed set (`compile.go:131`), so an
unchanged node recompiles to the same digest. Bundles are per-node with their own `Generation`
(`SignedBundle`, `store.go:188-196`).

## Changes (`internal/controller/compile.go` + store)
- In `CompileAndStage` (compile.go:211): per node, compute the new digest and compare against the
  node's CURRENT staged-or-served bundle digest. Equal → **skip staging that node** (its generation
  and served bundle stay put). Unequal, missing prior, or digest unreadable → stage normally
  (**fail open to a redeploy, never to a stale config**).
- **KEYSTONE-AWARE SKIP (review blocker — the load-bearing exception):** keystone credential
  ROTATION and FIRST-PIN change **zero bundle bytes** (the trust-list lives OUTSIDE bundles,
  compile.go:22-32; `stageManifest` REUSES the epoch when membership `node_id→{wgkey,digest}` is
  unchanged) yet REQUIRE a full re-stage: `PromoteStaged` returns `ErrNoStagedBundle` when nothing
  is staged **BEFORE** it copies the signed trust-list to the served slot (memstore.go:388-399), so
  a skipped deploy would leave the served trust-list signed under the rotated-away credential and
  `KeystoneRedeployRequired` stuck true — a fleet-stranding trap (the beta.4–6 class). Therefore:
  when `KeystoneRedeployRequired` is true OR keystone is enabled with NO served trust-list
  (first-pin), **disable the skip entirely for that stage** (full re-stage, every node, new
  generations) — automatically, no operator action. The preview (plan-6) must surface this state
  ("keystone rotation pending — a full redeploy will be staged") instead of claiming "0 changed".
- `StageResult` gains `UpdatedNodeIDs` / `UnchangedNodeIDs` (and counts) → surfaced by `HandleStage`
  so the panel can show "N updated, M unchanged". Audit (`appendStageAudit`, compile.go:405) records
  skips distinctly.
- **Zero-changed deploy:** short-circuit to a clean "nothing to deploy" StageResult — must NOT trip
  `ErrNoStagedBundle` (store.go:42-45) nor create an empty staged generation.
- **THE SEAM TO PROVE (before UX ships):** the keystone promote/serve path with MIXED generations —
  `PromoteStaged` (compile.go:434), the beta.5 served-vs-staged trust-list split, and the off-host
  manifest binding. The regenerated trust-list binds ALL nodes' digests OUT of the checksummed set;
  an unchanged node's entry is unchanged, so the new trust-list beside an old-generation bundle must
  verify cleanly on the agent. Regression tests in `internal/regression`:
  (a) recompile-unchanged-topology → ZERO new generations, zero golden churn;
  (b) one-node-changed → exactly that node re-staged; the OTHER node's served /config bytes are
      byte-identical before/after EXCEPT the trust-list, and its bundle still verifies against the
      new trust-list;
  (c) rekey-all → every CONNECTED node re-stages **via its PEERS' pubkey changes** (see the
      corrected invariant below); an ISOLATED node's digest is unchanged (skipped — benign, no
      tunnels; assert this explicitly);
  (d) keystone promote over a mixed set behaves (promote what's staged; served pairs stay atomic per
      `GetServedConfig`);
  (e) **keystone rotation / first-pin with an otherwise-unchanged topology → the skip is DISABLED
      (full re-stage), promote flips the served trust-list, `KeystoneRedeployRequired` clears** —
      the review-blocker case.
- **Corrected rekey invariant (review major):** a rekeyed node's OWN bundle digest does NOT change —
  its `[Interface] PrivateKey` is the `PRIVATEKEY_PLACEHOLDER` spliced from `/etc/wireguard/agent.key`
  at RUN time (script.go:805-833), and its own pubkey appears only in its PEERS' bundles + the
  trust-list. Rekey-all safety is therefore **peer-driven** (all connected nodes re-stage because
  their peers' pubkeys changed), NOT "new keys ⇒ new digest". Any FUTURE per-node rekey must
  force-stage the rekeyed node itself (route through the plan-6 Force machinery) or it partitions
  with its old key in-kernel. `redeploy_required`/reprovision flows: keystone-borne ones are covered
  by the keystone-aware skip above; content-borne ones re-stage naturally.

## Insertion-point markers
- **plan-5.5 (STOP-LOSS):** if the served-vs-staged split CANNOT safely serve an old-generation
  bundle beside the new trust-list (verification or atomicity breaks), **OR** the keystone-aware
  skip-disable cannot cleanly detect rotation/first-pin state (`KeystoneRedeployRequired` / absent
  served trust-list) at stage time, STOP — re-design with the owner before any skip ships. Do not
  work around it.
