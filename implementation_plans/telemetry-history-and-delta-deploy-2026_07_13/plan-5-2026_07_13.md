# plan-5 — staging skip-unchanged (per-node digest compare; keystone-seam proofs)

**Goal:** Deploy re-stages ONLY nodes whose bundle content changed; unchanged nodes keep their
current generation so agents see nothing new and never refresh. Agent (`cycle.go`) UNTOUCHED.

## Design (assessment recap)
Content identity = `hex(sha256(checksums.sha256))` — already the off-host-bound identity
(`compile.go:24-32`); `compiled_at` is already outside the checksummed set (`compile.go:131`), so an
unchanged node recompiles to the same digest. Bundles are per-node with their own `Generation`
(`SignedBundle`, `store.go:188-196`).

## Changes (`internal/controller/compile.go` + store)
- In `CompileAndStage` (compile.go:211): per node, compute the new digest and compare against the
  node's CURRENT staged-or-served bundle digest. Equal → **skip staging that node** (its generation
  and served bundle stay put). Unequal, missing prior, or digest unreadable → stage normally
  (**fail open to a redeploy, never to a stale config**).
- `StageResult` gains `UpdatedNodeIDs` / `UnchangedNodeIDs` (and counts) → surfaced by `HandleStage`
  so the panel can show "N updated, M unchanged". Audit (`appendStageAudit`, compile.go:405) records
  skips distinctly.
- **Zero-changed deploy:** short-circuit to a clean "nothing to deploy" StageResult — must NOT trip
  `ErrNoStagedBundle` (store.go:42-45) nor create an empty staged generation.
- **THE SEAM TO PROVE (before UX ships):** the keystone promote/serve path with MIXED generations —
  `PromoteStaged` (compile.go:434), the beta.5 served-vs-staged trust-list split, and the off-host
  manifest binding. The regenerated trust-list binds ALL nodes' digests OUT of the checksummed set;
  an unchanged node's entry is unchanged, so the new trust-list beside an old-generation bundle must
  verify cleanly on the agent. Regression tests in `internal/regression`:
  (a) recompile-unchanged-topology → ZERO new generations, zero golden churn;
  (b) one-node-changed → exactly that node re-staged; the OTHER node's served /config bytes are
      byte-identical before/after EXCEPT the trust-list, and its bundle still verifies against the
      new trust-list;
  (c) rekey → digest changes → staged (never skipped);
  (d) keystone promote over a mixed set behaves (promote what's staged; served pairs stay atomic per
      `GetServedConfig`).
- Rekey/`redeploy_required`/reprovision flows: verify they force content changes through naturally
  (new keys ⇒ new digest); anything that ISN'T content-borne needs the plan-6 Force, not a hack here.

## Verify + branch
Full Go suite (default+airgap, `-race` for controller) + gofmt; **assert zero golden diff** (no
rendered-artifact change in this plan). Branch `feat/deploy-skip-unchanged`.

## Insertion-point markers
- **plan-5.5 (STOP-LOSS):** if the served-vs-staged split CANNOT safely serve an old-generation

  owner before any skip ships. Do not work around it.
