## Implementation steps

### 1. FE mimic-deb type (P1 — do NOT reuse `AgentPin`)
In `controllerClient.ts`, add a distinct `MimicDebPin { asset; sha256; dkmsAsset?; dkmsSha256? }`
type and use it for the `mimicDebs` wire mapping only (leave `agentBins` on `AgentPin`). Map to/from
the Go `{asset,sha256,dkms_asset,dkms_sha256}` shape.

### 2. Row model + render (`MimicCatalogSettings.tsx`)
- Extend `DebRow` with `dkmsAsset`/`dkmsSha256`; seed from `initial.mimicDebs`.
- Row render: add a second (asset, sha256) input pair labeled "mimic-dkms module `.deb`" beneath the
  mimic pair, with `data-testid="mimic-dkms-asset-<id>"` / `…-sha-<id>` (e2e locates by testid).
- `filledDebs`: emit the companion fields when present. A row is a valid pin when the mimic asset +
  sha are filled; the dkms pair is required for split-package distros — surface a WARNING hint
  ("no module .deb pinned — install will fail on Debian 12 / Ubuntu 24.04") when the mimic pair is
  filled but the dkms pair is empty, matching the Go validator's disposition (error vs warn — mirror
  whatever plan-1's `validateMimicCatalog` chose; recommend: a warning here, hard-required only if the
  server rejects it).
- `validate`: apply `DEB_ASSET_RE`/`SHA256_RE` to the companion fields too (`:139-148`).

### 3. Discover pairs the `-dkms` asset (`mimicDiscover.ts` + component)
- `deriveKey`: still returns the `<codename>-<arch>` for a `mimic` asset; for a `mimic-dkms` asset,
  ALSO derive the same `<codename>-<arch>` (parse `^([a-z0-9]+)_mimic-dkms_.*_([a-z0-9]+)\.deb$`).
  Return a small discriminated result (e.g. `{key, slot: 'mimic'|'dkms'}`) OR keep `deriveKey`
  returning the label and add `deriveSlot(asset)`.
- Discover checklist: group a discovered `mimic` + `mimic-dkms` sharing a `<codename>-<arch>` into ONE
  addable row (mimic → `asset`, dkms → `dkmsAsset`). `handleAddSelected` writes both into the new
  `DebRow`. `collidingKeys`: a `mimic`+`dkms` pair on the same label is NOT a collision (only two
  same-slot labels collide) — update the guard + its test.

### 4. Assist fetches both sidecars (D6 — slot discriminator)
- `release_pins.go`: extend `releasePinAssetJSON` (`:200-203`) with an optional `slot`
  (`"mimic"|"dkms"`, default `mimic`) and key the response by `(key, slot)` — e.g. response
  `pins[key] = {asset,sha256,dkms_asset,dkms_sha256}` or a `pins[key][slot]` shape. Fetch each
  requested `(key, asset)` sidecar independently (loop `:352-367`); `fetchSidecar` (`:395-419`)
  already re-checks `sha256HexPattern` — ensure an empty/whitespace/HTML body yields NO pin (not a
  pin with `sha256:""`).
- `MimicCatalogSettings.tsx` `handleAssist`: send both the mimic and dkms assets per row (with
  slots); fill both shas. **Fix the empty-SHA silent-ignore** (`:182-190`): if a returned pin's
  `sha256` is empty/does not match `SHA256_RE`, treat it as a MISS — set the row note
  ("Couldn't fetch a sidecar for `<asset>` — enter the SHA-256 manually"), never write a blank sha.
- Investigate the user-reported "essentially empty" case: confirm whether the backend ever returns a
  pin with `sha256:""` (vs. no pin). If so, make the backend omit the pin entirely on a
  non-64-hex sidecar; the UI then shows the existing miss note. Add a `release_pins_test.go` case for
  an empty/garbage sidecar → no pin.

### 5. Go `validateMimicCatalog` companion checks (`handler_bootstrap.go:268-303`)
In the per-row loop (`:291-301`), if `DKMSAsset`/`DKMSSHA256` are set, validate them with
`debAssetPattern`/`sha256HexPattern`; if one is set without the other, reject
(`incompletePin`-style). Mirror in the TS `validate` + any `validator.ts` catalog mirror. Add i18n
keys (`en.ts`+`zh.ts`) for the new dkms labels/hints (gated by `TestI18nCatalogSync`/`Parity`).
