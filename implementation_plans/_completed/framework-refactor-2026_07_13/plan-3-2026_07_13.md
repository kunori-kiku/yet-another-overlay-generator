# plan-3 ‚Äî WASM add-alongside + PERMANENT gate

**Goal:** Run the pure Go pipeline IN the browser next to the TS compiler, proven byte-equal over the
full corpus, **with nothing deleted**. The permanent WASM-vs-golden gate is the fix for the judges' #1
concern ‚Äî never trust "same-source ‚áí same-target," pin it forever (this repo's own philosophy).

**Prerequisites:** plan-1 (`localcompile` is the heal-inclusive single entry), plan-0 (the ratchet /
hygiene baseline). **This is the strategic-bet gate ‚Äî Phases 0‚Äì2 stand alone if the owner stops here.**

## Design (assessment recap)
**Verified feasible:** `GOOS=js GOARCH=wasm go build ./internal/{localcompile,compiler,renderer,validator,
allocator,artifacts,normalize,render,model,naming,linkid,allocconst}` succeeds on the current tree.
`internal/render` imports `wgctrl/wgtypes` (types only ‚Äî wasm-compatible); the keygen `crypto/ecdh` seam
+ `keygen_equivalence_test.go` already exist. Output is deterministic integer/string ‚Äî no floats, no
timestamps ‚Äî so `GOARCH=wasm` map-order/float/time quirks cannot silently reach output, and the three-way
gate catches any quirk BEFORE any deletion.

## Changes
- New `cmd/wasm` ‚Äî `GOOS=js GOARCH=wasm` main; a `syscall/js` shim exposing `localcompile.Compile` +
  `Validate` / `Export` / `DeployScript`; emits `web/yaog.wasm`. Add the build to CI + Vite.
- New `frontend/src/wasm/wasmEngine.ts` ‚Äî loads `web/yaog.wasm` + a version-pinned `wasm_exec.js`; exposes
  `compile/validate/export/deployScript`; wired behind `VITE_YAOG_LOCAL_ENGINE='wasm'` (via plan-0's
  `deployMode.ts`). TS stays the DEFAULT this phase.
- **The permanent gate** ‚Äî a headless CI job that EXECUTES the built `yaog.wasm` against the Go golden
  corpus AND asserts **three-way equality (WASM == Go golden == TS output)** over the full
  SUCCESS/render corpus (the validator + `apierr` FAIL/verdict corpus is a Go-side conformance test,
  re-homed into `localcompile` by plan-5 ‚Äî it is not part of the WASM gate). Pin `wasm_exec.js` to the
  building toolchain; fail the build on drift. Register it as a required check (update branch protection
  in the same PR, per the repo rule).

## Verify + branch
Full Go + FE suites; the new three-way gate green over the whole corpus; `GOOS=js GOARCH=wasm go build`
in CI; the `wasm_exec.js`‚Üîtoolchain pin asserted. Branch `refactor/plan-3-wasm-alongside`.

## Tests produced
- The permanent **WASM-vs-golden three-way gate** ‚Äî **perpetual**, a REQUIRED check. Guards parity by
  execution. Retirement: never (it becomes the sole compiler oracle after plan-5).
- A `wasmEngine.ts` load/roundtrip test ‚Äî perpetual.

## Invariants at risk
- **[1] parity by execution + permanent gate:** the gate MUST be green + registered before plan-4/5; do
  NOT proceed to the flip on a red or flaky gate.
- **[2] deterministic output:** if the gate finds ANY WASM-vs-Go byte difference, that is a
  `GOARCH=wasm` quirk to characterize (plan-3.5), not to `-update` away.
- **[4] port authority / [6] custody:** WASM local mode is design-PREVIEW only (never deploys/stages);
  WASM keygen uses the `crypto/ecdh` seam, public keys only; the shim adds ZERO Go deps.

## Stop-loss
**plan-3.5** if a determinism quirk surfaces in the three-way gate ‚Äî characterize + pin it before the
flip; NEVER delete the TS twin (plan-5) until the gate is green AND soaked (plan-4).

## Out of scope
Flipping the default (plan-4); deleting the TS twin or `conformance/` (plan-5); the WASM live-preview
latency characterization (an accepted open task ‚Äî mitigate with debounce; controller preview is
server-side).
