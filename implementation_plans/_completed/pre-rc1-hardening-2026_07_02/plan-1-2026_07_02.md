## Goal
A per-edge NAT override (`endpoint_host`/`endpoint_port`) is honored deterministically in the compiled
WireGuard config; eliminate the silent fallback to the peer's plain public IP ("direct to the other
IP"). The "NAT override active" badge and the compiled `Endpoint` line must never disagree.

## The confirmed defect cluster (from investigation 2026-07-02)
The override host is the linchpin. When the host is absent while a port override is set, the compiler
drops the override (forward `if edge.EndpointHost != ""` is false → no endpoint; reverse → the
`PublicEndpoints[0]` fallback), while the badge (port-only) still says "NAT override active":
1. **Compiler forward path** requires `edge.endpoint_host`, ignores a port-only override, and has NO
   public-endpoint fallback (unlike the reverse). A "set just the NAT port" flow silently drops.
2. **`reconcileEdgeEndpoints`** clears host+port when the target's public endpoint is renamed/removed —
   can leave the override gone while the operator believes it active.
3. **The "unset host" dropdown** (`EdgeEditor.tsx:205-209`) clears host but KEEPS the port → port-only
   inconsistent state.

## Implementation steps

### Step 1 — Reproduce (BEFORE any fix), then record the trigger
Construct minimal topology fixtures reproducing "direct to the other IP" and drive BOTH compilers:
- **(a) port-only override:** an edge with `endpoint_port>0`, `endpoint_host=""`, target has a
  `public_endpoints[0]`. Expect (current, buggy): forward `Endpoint` empty / reverse dials the plain
  public IP at the ALLOCATED port, ignoring the override port.
- **(b) reverse-only public node:** a single directed edge A→B where A is public; assert B's config for
  A uses `A.PublicEndpoints[0]` at the allocated port (the "direct" path) — characterize whether the
  operator expected the override here.
- **(c) reconcile clear:** set an override, then remove the target's public endpoint (drive
  `reconcileEdgeEndpoints`); assert the override is gone.
- If the owner supplies the affected topology/link JSON, add it verbatim as the primary fixture.

Run these through `internal/compiler` (Go) and `frontend/src/compiler` (TS) and capture the actual
`Endpoint` lines. **RECORD in `outline.md` §6 which mechanism(s) reproduce, and the chosen semantics
(resolve-from-public-endpoint [recommended] vs require-explicit-host).** Do not write the fix until
this is recorded.

### Step 2 — Compiler fix (Go + TS, byte-identical), per the recorded decision
Recommended semantics (confirm in Step 1): the forward endpoint resolves as
- explicit `edge.endpoint_host` (current), else
- when the target node is publicly reachable, `toNode.PublicEndpoints[0].Host` (NEW forward fallback,
  symmetric with the reverse `:897`),
- with the port = `edge.endpoint_port` when >0 (the operator override — a port-only override now
  resolves a host), else the allocated listen port.

Do NOT auto-stamp `endpoint_port` anywhere (backend-is-sole-port-authority). Change ONLY the
`Endpoint`-line derivation; the allocation pins (transit IPs, ports, link-locals) MUST be byte-identical
(allocation-stability). Apply the same change to `peers.ts` character-for-character (guard against a
literal backtick in the TS template). Keep the reverse `PublicEndpoints[0]` fallback but ensure it
never shadows an explicit override.

### Step 3 — Frontend store host/port coupling
- `reconcileEdgeEndpoints`: when clearing on a removed public endpoint, keep host+port coupled (either
  keep both or clear both — never leave a port with no host). Prefer clearing both (current behavior on
  `newHost===null`) but ensure the rename branch (`:548`) does not orphan a port.
- The "unset host" dropdown (`EdgeEditor.tsx:205-209`): also clear `endpoint_port` (keep host/port
  coupled) OR make the badge require a resolvable host. Choose the coupling that matches the Step-2
  compiler semantics so the badge and the compiled output agree.

### Step 4 — Validation (deployable-config)
If the chosen semantics leave any case where a port override cannot resolve a host (no edge host, no
target public endpoint), emit a clean validator error rather than a silent drop. If this needs a new
`validator.Code`, regen the drift manifest + add the `error.<code>` i18n (en+zh) + mirror in
`validator.ts`. (If the recommended resolve-from-public-endpoint semantics make every port-only
override resolvable when the target is public, the error only fires for a genuinely un-dialable
override — still worth coding.)

### Step 5 — Regenerate goldens + drift; add regression fixtures
- `go test ./internal/conformance/ ./internal/localcompile/ -update`; review the diff — the ONLY
  expected byte churn is `Endpoint` lines for edges that gained a resolved host (intended); pins must be
  unchanged. Commit the regenerated goldens + `drift_manifest.json`.
- Add the Step-1 fixtures as permanent conformance fixtures.
