## Goal
The generated bootstrap script verifies the downloaded agent binary against the controller's already-held
`AgentBins` SHA-256 pin before installing it — so a MITM/compromised-mirror cannot ship a root binary at
first contact.

## Implementation steps

### Step 1 — thread the pins into the script
Add an `AgentBins` param to `renderBootstrapScript` (`HandleBootstrap` already has `cs`). Bake per-
configured-arch shell-safe assignments via `shQuote` (mirror the `OP_FLAGS` accumulator style), e.g.
`AGENT_SHA_linux_amd64='<hex>'` / `AGENT_ASSET_linux_amd64='<asset>'`. Keep `RELEASE_BASE`/`GH_PROXY`
as today.

### Step 2 — verify before install
In `bootstrapScriptBody`, after `uname -m`→arch mapping, resolve `key="linux-$arch"`, look up the baked
asset+sha. When a pin exists: download `${GH_PROXY}${RELEASE_BASE}/${asset}` to a `mktemp`, then
`echo "${sha}  ${tmp_bin}" | sha256sum -c -` (fail-closed) BEFORE `install -m 0755`. Keep the existing
arch→default-asset (`yaog-agent-linux-<arch>`) as the fallback asset name when the operator did not
override it. The pin makes plain-`http` transport integrity-independent, so `--proto '=https,http'` need
not change.

### Step 3 — empty-`AgentBins` policy
Verify-when-pinned (fail-closed on mismatch); when NO pin exists for the arch, emit a loud
`echo ">> WARNING: no SHA-256 pin configured for $key; binary integrity NOT verified" >&2` and proceed
(preserves the "just set a release URL" bring-up — the codebase's opt-in/byte-identical-when-unset
convention). Note the stricter alternative (hard-fail unpinned) for the owner in the PR description.
