## Goal (milestone)

Meet the hosted multi-tenant SaaS bar: many tenants on one service, cryptographic tenant isolation,
signing keys out of the controller process, and a deploy gate a live breach can't abuse for the
catastrophic cases.

## Scope (to be decomposed when detailed)

- **Per-tenant KMS config-signing** (Tier B routine): move the Phase-0 signing key into per-tenant
  KMS **sign-only, non-exportable** handles (not in the controller process, not in the DB). Agents
  pin a per-tenant **trust set** (rotation = add-new/overlap/retire, no fleet re-pin).
- **Hardware-signed membership** (Tier A): admit/evict/rekey require a **human hardware-key signature
  over the change content** (content-bound; agent verifies against a pinned anchor). The controller
  has no autonomous capability to change membership — closes the catastrophic slice of the live-breach
  window. (PIV/PGP token signs content directly; WebAuthn signs a content-committing challenge.)
- **Structural tenant isolation:** `tenant_id` from the authenticated principal only, single
  chokepoint; `/config`/`/poll` return only the caller's node; per-tenant signing keys mean a
  cross-tenant bundle cannot verify. **Cross-tenant access CI gate** (perpetual).
- **Operator auth:** OIDC login + tenant RBAC.
- **Stage → Promote:** stage shows the exact per-node diff (byte-stable bundles); Tier-B promote
  requires a **WebAuthn/FIDO step-up**, with a **warned, explicitly-acknowledged one-click fallback**
  for FIDO-incompatible environments (+ mandatory loud promote audit-alarms); Tier-A membership
  changes always require the hardware signature (no fallback); instant rollback by promoting a prior
  signed version.
- **Supply chain:** reproducible agent builds, hash-pinned deps, **release-signing key distinct from
  config-signing, under dual control**, verify-before-update, opt-in staged updates, SBOM/provenance.
- **Data-at-rest:** standard at-rest for v1 (DB holds public keys + topology only); per-tenant
  envelope encryption is a later upgrade.

## Hazards / gates (milestone)

The single biggest residual is the **live-breach signing+promote window for Tier-B routine config**
(membership is hardware-gated, so it's closed there) — bounded by KMS (no offline forgery), step-up
promote, anti-rollback, instant rollback, audit alarms, and the air-gap escape. Gate: cross-tenant
isolation CI gate + a tabletop/red-team of the breach scenarios + real-host multi-tenant smoke.
Stop-loss: ship single-tenant GA first if multi-tenant isolation review isn't clean; KMS/hardware
signing can land before OIDC/RBAC if needed.

## Out of scope

Everything in the program's out-of-scope list (reconciler, NAT traversal, embedding Headscale/
Netbird, browser keygen, compiler-purity changes, commerce, MagicDNS/ACL). Per-tenant envelope
encryption deferred. Billing/quota explicitly excluded — isolation is in scope, commerce is not.
