## Goal
Fix a confirmed HIGH availability defect: the self-update **reconcile** compares the running
`BuildVersion` to the stored breadcrumb `To` with **exact string (in)equality**, while the self-test
that STORED `To` used the **semver comparator**. A valid `v`-less operator target (`"2.0.0"` vs the
released `BuildVersion "v2.0.0"`) passes the swap to a healthy new binary, then reconcile sees
`"v2.0.0" != "2.0.0"` and takes the "swap never took effect" branch — so the floor never advances, the
breadcrumb persists, the in-flight guard blocks ALL future updates, and abandon records the running
version. The self-update channel is **permanently wedged fleet-wide**. Also address the broader T5
durability cluster (brick-bound hardening) the sweep flagged.

## Implementation steps
1. Route the reconcile/finalize/rollback version equality at `selfupdate.go:338/377/414` through the
   SAME `compareVersions()` semantics the self-test uses at `:264` (a shared `sameVersion(a,b) bool`
   helper that strips a leading `v` on both sides), so a `v`-less target and a `v`-prefixed BuildVersion
   are treated as equal. Keep `To: cat.Version` raw (the catalog value) — normalize at compare time,
   not at store time (back-compat with breadcrumbs already on disk).
2. Add a regression test reproducing the wedge: breadcrumb `To:"2.0.0"`, `BuildVersion:"v2.0.0"` → assert
   promote health-confirms + the floor advances + the in-flight guard clears (today it wedges).
3. Apply the remaining T5 brick-bound hardening items the sweep listed (read ASSESSMENT.md T5 members) —
   ONLY where they are real gaps, not churn on the sound breadcrumb/rollback design.
