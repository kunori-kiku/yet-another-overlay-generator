# plan-6 — WebAuthn User-Verification enforcement (HIGH security; owner decision gate)

- **Subject:** post-refactor-debt-paydown-2026_07_14 · **Tier:** 1 (correctness & security fixes)
- **Status:** merged 2026-07-15, then superseded before release · **Branch:** `fix/webauthn-enforce-uv`
- **Reads from specs:** `panel-auth`, `keystone-trustlist`

> **Historical plan:** the blanket runtime/node assertion gate described below was merged but never
> released. Before the still-uncut rc.7, it was replaced with server-verified UV proof only when a new
> login or keystone browser credential is enrolled, avoiding a retroactive requirement that could lock
> out existing operators or make upgraded nodes reject their currently served manifest. See `CHANGELOG.md` Unreleased and
> `docs/spec/controller/operator-auth.md` for the current contract; no fleet re-sign is required.

## Prerequisites
None code-wise, but plan-6 **STOPS for an owner decision** before hard-enforcing (see the gate below).

## Goal
Close a confirmed HIGH auth-downgrade: the shared assertion verifier checks ONLY the User-Present bit
(`0x01`) and **never** the User-Verified bit (`0x04`), even though both ceremonies request
`userVerification:'required'` on the client. The server is the sole enforcement authority, so a
User-Present-only assertion (a PIN-less/biometric-less security key, or a tampered/malicious client
requesting UV=discouraged) mints a full operator session — the declared "passwordless + user-verified"
factor silently collapses to possession-only. The one verifier backs operator **login + 2FA + keystone
off-host signing**, so a single gate fixes all three.

## Owner decision gate (per outline decision 3)
Before hard-enforcing UV, CONFIRM with the owner: **does any enrolled operator authenticator lack UV**
(a bare security key with no PIN/biometric)? Synced passkeys (Touch ID, Windows Hello, phone) DO perform
UV when required, so for those the fix is transparent.
- If **no** UV-incapable authenticator is enrolled → hard-enforce (fail-closed on missing `0x04`).
- If **yes** → do NOT blanket-lock-out; open **plan-6.5** for enforce-on-UV-capable + a re-enrollment
  migration path. (This is why the gap-pass dispositioned UV "separate-security-audit" — the code fix is
  trivial, the rollout needs care.)

  ES256/EdDSA verify; the signature-counter skip is correct).

## Implementation steps
1. Add `const flagUserVerified = 0x04` and, in `verifyAssertion` after the UP check, require
   `authData[32]&flagUserVerified != 0` (fail-closed) — a single gate so login, 2FA, and keystone
   signing all inherit it. (The clients already pin `userVerification:'required'`.)
2. Update `verify_assertion_test.go:27` to build authData with `UP|UV` (`0x05`) and add a NEGATIVE test:
   a UP-only assertion is REJECTED.
3. If the owner gate says an enrolled authenticator is UV-incapable → STOP; do not merge the blanket
   gate; draft plan-6.5 (conditional enforcement keyed on the credential's enrolled UV capability + a
   re-enroll prompt).

## Tests produced

## Verification

## Definition of done
- The server enforces UV (fail-closed) for the UV=required ceremonies; a UP-only assertion is rejected;
  positive + negative tests pin it; the owner gate resolved (hard-enforce, or plan-6.5 opened). Review →
  fix → re-review clean → CI green.

## Out of scope
The signature-counter (correctly skipped). The rest of the ceremony (sound). Any change to the challenge
minting / RPID / origin logic.
