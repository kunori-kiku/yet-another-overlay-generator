> Part of the pre-rc.1 program — see outline.md for mission, principles, and full sequencing.

## Prerequisites

- **plan-11 (2.2 — app-shell mobile adaptation) MUST land first.** 2.2 owns the off-canvas Sidebar drawer, the Topbar hamburger, the `mobileNavOpen` uiStore field, and **exports** the two shared primitives this plan may consume: `useMediaQuery` (with the single shared breakpoint constant `LG_QUERY = '(max-width: 1023.98px)'`) and the reusable `Drawer` component. 2.1 reclaims usable page width *only because* the drawer exists, and (if the NodeRegistry reflow needs a runtime branch) imports `useMediaQuery`/`LG_QUERY` from 2.2. **Order: plan-11 → plan-10.** This plan builds NONE of those primitives and must not re-decide the breakpoint.
- **plan-1 (1.1 — hygiene normalization) SHOULD land first (or be explicitly coordinated).** 1.1 Englishizes the CJK comments in `deploy/NodeRegistry` (one of 32 frontend files) in a byte-preserving pass. This plan adds a net-new card block adjacent to that code; landing after 1.1 avoids comment-debt churn and overlapping diffs in the same file. This plan adds only English comments and does not touch the existing CJK.
- **Program spine:** Subject 2 (plans 10–12) runs after Subject 1 (plans 1–9) and after v2.0.0-beta.8 (B1/F1/C1/S4/S5/S6 already merged). This plan touches no compiler/model/Go code, so it is independent of the Go↔TS conformance harness (plan-5) and the TS-migration isolation discipline — `drift_risk` stays FALSE.
- **Sibling plan-12 (2.3 — canvas small-screen)** also adopts `lg`=1024 + 2.2's `useMediaQuery`. No file overlap with this plan.

## Goal

Give the YAOG **operator pages** a responsive breakpoint pass so the fleet/operator surfaces are **usable on a phone** (≈360px) and tablet (≈768px) without regressing the desktop layout. For the six operator pages (Overview, Fleet, Deploy, Security, Settings, Fleet-node-detail) plus the Audit view: collapse page padding (`p-6` → `p-3 sm:p-6`), collapse the OverviewPage stat grids (`grid-cols-3` → `grid-cols-1 sm:grid-cols-3`), reflow the 8-column NodeRegistry fleet table into stacked cards below `lg`, bring sub-44px action tap targets up to the touch guideline, and opportunistically swap hardcoded `bg-gray-*` to the shell's theme tokens **only on lines an in-scope edit already crosses**. This plan owns ONLY the per-page tier and **consumes** 2.2's `useMediaQuery`/`Drawer`; it builds no drawer/hamburger/`mobileNavOpen` and touches no shell file (`Sidebar.tsx`, `Topbar.tsx`, `Shell.tsx`, `uiStore.ts`) and no React Flow canvas file.

## Implementation steps

> Phasing starts at Phase 1 because the former "Phase 1 — drawer" was deleted (it moved to plan-11/2.2). This plan begins only after 2.2's drawer + `useMediaQuery`/`Drawer` exports land.
>
> **Breakpoint decision (FINAL — not an open question).** The table↔card crossover and the sidebar-dock threshold use **`lg` = 1024px**; the OverviewPage stat-grid collapse uses **`sm` = 640px**. plans 10, 11, 12 MUST all adopt the identical `lg`=1024 boundary. Because the layout work here is pure Tailwind (`lg:`/`sm:` utility classes), this plan does not itself read a JS media query for layout. If the NodeRegistry presentation must branch at runtime (table vs cards), it consumes 2.2's `useMediaQuery(LG_QUERY)` so the JS query and the `lg:` classes can never drift. **Default is the CSS-only path (`hidden lg:block` / `lg:hidden`) which needs no import.**
>
> **Mobile-first pattern:** bare Tailwind class = phone; `sm:`/`lg:` = wider. No custom breakpoints, so `index.css` stays untouched.
> **Minimal-diff theme tokens:** where a responsive edit already rewrites a `className` containing `bg-gray-900`/`bg-gray-800`/`border-gray-700`, swap to `var(--surface)`/`var(--surface-elevated)`/`var(--hairline)`/`var(--content)` *in that same string*; never open a line solely to swap a color (the full sweep is OUT).
> **English-only new comments:** any comment this plan adds (notably on the NodeRegistry shared-cells array) is English; do not translate the file's existing CJK (1.1's remit).

## Tests produced by this plan

The in-PR merge gate is `npm run build` (tsc -b) + `npm run lint` + the Phase 4 manual responsive smoke at two widths.

- **Structural regression guard (no test file):** the shared per-node **descriptor-array spine** in `NodeRegistry.tsx` (step 5) is the in-code mitigation against table↔card cell-data drift — one source list feeds both presentations, so adding/removing a column edits one array, not two JSX trees. This pulls toward 1.2's single-source-of-truth bar; it is reinforced by, not dependent on, the English code comment.
  - **Lifetime:** perpetual (it is production structure, not a test).
  - **Guards:** table and card presentations never diverge in which per-node fields they show.
  - **Retirement trigger:** n/a (production code; superseded only by a future redesign of NodeRegistry).
  - **Retirement destination:** n/a.

  - **Lifetime:** subject-scoped (the manual checklist; superseded by 3.5's automated corpus).
