## Goal

Decide and implement how the React Flow design canvas (`/design` route) behaves on small (phone / narrow-tablet) viewports. The locked decision: a graceful breakpoint GATE — a "design on a larger screen" interstitial — layered on top of a **read-only pan/zoom preview** of the existing canvas, rather than chasing full touch-edit parity or shipping a bare hard block. Below `lg` (1024px): React Flow edit interactions are turned OFF (`nodesDraggable`/`nodesConnectable`/`elementsSelectable={false}`) while pan/zoom keep their defaults, the editing chrome (`CanvasToolbar`, the `w-72` lists drawer, the `w-80` `DesignAside`, the `h-40` validation footer) is not rendered so the read-only canvas gets the full narrow width, and a dismissible `CanvasGate` explains that editing requires a larger screen. At/above `lg` the existing desktop layout is byte-for-byte behaviorally unchanged. A phone operator can VIEW/inspect a topology (genuinely useful and nearly free, since React Flow's touch pan/zoom already works with no props set) but is steered to a larger screen to edit.

## Implementation steps

Work on a feature branch (NOT `main`). The diff is confined to `frontend/src/`.

**Phase 1 — Responsive primitive.**

1. Create `frontend/src/lib/useMediaQuery.ts`:
   - `export function useMediaQuery(query: string): boolean` — `useState` initialized synchronously to `window.matchMedia(query).matches` (client-only SPA, no SSR mismatch, same assumption as `ThemeProvider.tsx:7-8`), plus a `useEffect` subscribing to `media.addEventListener('change', …)` with `removeEventListener` cleanup, directly mirroring `ThemeProvider.tsx:26-33`. The effect must re-subscribe when `query` changes.
   - Add a tiny named helper `export function useIsDesktop(): boolean` that calls `useMediaQuery('(min-width: 1024px)')` so the `lg` threshold has exactly ONE definition shared with any Tailwind `lg:` utilities (Tailwind v4 default `lg` = 1024). This is the single source of truth the sibling plan-11 also imports.

**Phase 2 — Canvas read-only interaction matrix** (`frontend/src/components/canvas/TopologyCanvas.tsx`).

2. Add an `editable: boolean` prop to `TopologyCanvas` **defaulting to `true`** (so any caller that omits it keeps current behavior — the desktop regression guard, R1). Prefer the prop over calling `useIsDesktop()` internally so `DesignPage` owns the breakpoint and the canvas stays presentational.
3. On the `<ReactFlow>` element (`:565-585`), when `!editable` set `nodesDraggable={false}`, `nodesConnectable={false}`, `elementsSelectable={false}`. Do NOT set `panOnDrag`/`zoomOnPinch` — leave pan/zoom at defaults so the read-only preview still pans/pinches. Leave `fitView`/`fitViewOptions` (`:582-583`) intact so the read-only view still frames the graph.
4. Add early-returns to the store mutators so a stray change can NEVER mutate the store from the read-only view (belt-and-braces; R2): guard `handleNodesChange`'s `remove` branch (`:334-337`) and `handleEdgesChange`'s `remove` branch (`:351-353`) — when `!editable`, do not call `removeTopoNode`/`removeTopoEdge`. Guard `onConnect`/`onConnectStart`/`onConnectEnd` (`:485-522`) to be inert when `!editable`. Keep `onNodeClick`/`onEdgeClick`/`onPaneClick` (`:524-542`) working (selection-for-inspection is harmless and `DesignAside` is hidden below `lg` anyway).
5. Hide the auto-layout + show-interfaces `<Panel>` (`:589-605`) when `!editable` (auto-layout mutates positions; the toolbar is fiddly on a phone — hide the whole `<Panel>` for simplicity). KEEP `<Controls>` (`:587`, pan/zoom buttons — useful read-only) and `<MiniMap>` (`:606-618`).

**Phase 3 — DesignPage layout branch + gate.**

6. In `frontend/src/components/pages/DesignPage.tsx`, call `useIsDesktop()`.
   - `>= lg`: render the **existing JSX unchanged** (`DesignPage.tsx:19-41`) — guarantees zero desktop regression (R1).
   - `< lg`: render a small-screen branch — a full-bleed `<TopologyCanvas editable={false} />` plus the new `CanvasGate`; do NOT mount `CanvasToolbar` (`:21`), the lists drawer `aside` (`:23-31`), `DesignAside` (`:35`), or the `h-40` footer (`:37-39`).
7. Create `frontend/src/components/design/CanvasGate.tsx`: a dismissible banner/overlay reusing the modal idiom from `CanvasToolbar.tsx:133-134` (`fixed inset-0 z-50 grid place-items-center bg-black/50 p-4`, `w-full max-w-md … rounded-lg … bg-gray-800 p-5` card) with an i18n heading + body ("Topology editing needs a larger screen") and actions:
   - "View read-only" — a session-only (non-persisted) local `useState` dismiss flag that drops the overlay to reveal the read-only canvas. It NEVER re-enables edit interactions, preserving the no-touch-edit guarantee.
   - (optional) "Open menu" — a conditional/affordance to open the shell hamburger menu, rendered ONLY if plan-11's shell drawer has landed; the gate must be fully self-contained without it.
   - Keep it presentational; no store writes beyond the local dismiss flag.

**Phase 4 — i18n** (parity gate).

8. Add a `canvasGate.*` group to `frontend/src/i18n/messages/en.ts` near the `canvasToolbar.*` block (`:213-223`): `canvasGate.title`, `canvasGate.body`, `canvasGate.viewReadOnly`, `canvasGate.readOnlyBadge`, and (if used) `canvasGate.openMenu`. Add the IDENTICAL key set with translations to `frontend/src/i18n/messages/zh.ts` — the `Record<keyof typeof en, string>` guard at `zh.ts:7` will fail `tsc -b` if any key is missing (this is the enforced parity gate, R4). Explicit owner brief; the edit affordances (`w-56` forms `CanvasToolbar.tsx:69,72`, connect-drag `onConnect`/`onConnectStart`/`onConnectEnd`, `w-80` aside editors `DesignAside.tsx:18`) are not phone-shaped and would be a large, risky rework against the Subject 1 PR stream.
- **The app-shell off-canvas drawer + Topbar hamburger** (plan-11 / 2.2) and **the operator-page (Fleet/Deploy/Nodes/Audit/Overview/Settings) responsive pass + NodeRegistry card reflow + tap-target bumps** (plan-10 / 2.1) — the sibling Subject 2 plans. This plan depends on / coordinates with them (shared breakpoint + the `useMediaQuery` hook this plan creates) but does not own them. The four un-yielding columns the canvas must coexist with — the global shell sidebar (`shell/Sidebar.tsx`, `shell/Shell.tsx`, owned by plan-11) plus the Design route's own `w-72` drawer / `w-80` aside / `h-40` footer — are addressed for the canvas branch here but the shell column itself is plan-11's.
- **Theme-token migration** of `bg-gray-900`/`bg-gray-800` on the Design surfaces — a cheap rider for the sibling pass, not a phone blocker, not this plan.
- **Any backend, compiler, model, or `controllerClient` change. No new npm dependency** (React Flow + `matchMedia` are already in use).
- **An automated frontend test** — there is no test runner; the future vitest unit test for `useMediaQuery` belongs to the FE-test-runner plan (noted above).
