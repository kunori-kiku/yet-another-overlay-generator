## Goal

After Subjects 1+3 have mechanized everything a harness/E2E/netns tier can mechanize,
enumerate — with exact commands and binary pass/fail predicates — the **irreducible**
real-hardware/firmware/kernel manual smokes the owner MUST run before rc.1, and feed that
triage into the rc.1 gate ledger (plan-22 criterion C1). The owed set shrinks from the
**eight** fuzzy smokes in `STATUS.md:58-89` to the **smallest** set of sharply-scoped
real-silicon/firmware/kernel runs that no in-process or netns test can synthesize honestly,
via a **three-stage funnel**: subtract the Go↔TS conformance harness (1.5, bucket A), the
browser E2E rig (3.1–3.5, bucket B), and the real-tunnel netns tier (3.6, bucket 3.6-netns).
The decisive new subtractor is 3.6: because it brings up the generated WireGuard+Babel+SNAT
data plane on a **real Linux kernel** and asserts handshake/route/ping/SNAT/C3, most of the
old "real multi-host NAT" smoke is now netns-covered, collapsing the residue to **two**
genuinely-irreducible categories plus **one** narrow NAT-box residual: **C1** a real WebAuthn
authenticator device (firmware/UX); **C3** mimic's eBPF/DKMS/XDP module on a real ≥6.1 kernel
(the one data-plane leg 3.6 explicitly excludes); and the **shrunk C2** — real-NAT-box
endpoint-rewrite survival ONLY, a single property a single-host netns cannot synthesize. This
is an **enumeration/documentation plan — no production code changes**; all code references are
read-only confirmations. Its value is honesty: a defensible line between what a harness/netns
CAN replace (and therefore must, in Subjects 1+3) and what only real silicon/firmware/kernel
can prove.
