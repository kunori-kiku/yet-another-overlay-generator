# plan-3 — history query API (server-side aggregation, operator-gated)

**Goal:** the panel can ask for any node's resource history at any granularity; the server
aggregates raw samples into bounded bucket series.

## Changes
- `GET /api/nodes/{id}/telemetry/history?from=<RFC3339>&to=<RFC3339>&step=<duration>` on the
  **operator mux** (same auth gate as the other operator routes; NEVER the agent/anonymous muxes).
- Response: `{ step, buckets: [{ t, cpu_pct?, load1, mem_used_pct? … } ] }` where each metric bucket
  carries `avg/min/max` over the raw samples in the window (aggregate ALL metrics per bucket in one
  pass — one endpoint, not per-metric; the chart picks which series to draw).
- Guards: clamp `step` to ≥ the heartbeat cadence and the response to ≤ ~1000 buckets (widen step
  server-side if needed, echo the effective step); `from/to` sanity (to > from, bounded range);
  unknown node → 404; history disabled (cap=0) → 200 with empty buckets + a flag.
- Wire through Store: a `QueryTelemetryHistory(ctx, tenant, nodeID, from, to)` store method
  (ring + lazy-loaded tail from plan-2); aggregation lives in the handler layer (pure, table-tested).

## Tests
- Pure aggregation table tests (bucket boundaries, avg/min/max, gaps stay gaps — buckets with zero
  samples are OMITTED, never zero-filled; cpu_pct absent-aware).
- Handler tests: auth-gated (401 anonymous), 404, clamping (tiny step → widened + echoed), empty.
