# plan-5 — Per-node update-status surface (Fleet + Node detail) + opt-in poll

See `outline.md`. Milestone: observability surface. **Prerequisites:** plan-1 (`done`, the `in_rollout`
flag), plan-2 (`done`, the settings carry `targetAgentVersion` + canary set).

## Goal
Add a derived per-node **update-status chip** to the Fleet registry + a matching field on the node detail
page, plus an opt-in auto-poll so canary→applied transitions appear live. The chip is computed by a pure
`deriveUpdateState(node, settings)`; no new agent/wire field is added.

## Implementation steps

1. **Carry the new node field** (`frontend/src/types/controller.ts` + the node mapping in
   `controllerClient.ts`): add `inRollout: boolean` from plan-1's nodeJSON (safe default `false`). The
   target is read from `settings.targetAgentVersion`, NOT echoed per-node (decisions log).

2. **`deriveUpdateState(node, settings)`** (pure, co-located, mirror `isDrifting`): returns one of
   `'off' | 'not-targeted' | 'pending' | 'applying' | 'applied' | 'failed' | 'stale'`. Evaluate in THIS
   order — the `lastHealth` markers are SUBSTRINGS, quoted verbatim from `selfupdate.go`:
   - `settings.targetAgentVersion === ''` → `off` (no chip — the safety contract).
   - not `node.inRollout` (server-computed, plan-1) → `not-targeted`.
   - `node.lastHealth.includes(' abandoned:')` → `failed` (best-effort: the marker is TRANSIENT,
     overwritten by the next apply report — see outline Principle 7 + plan-5.5; a permanently-failed
     canary degrades to `pending`).
   - `node.lastHealth.includes('health-confirmed (probationary)')` → `applying`.
   - `node.lastHealth.startsWith('self-updated to ')` → `applied`.
   - no health marker present: compare reported `agentVersion` vs `settings.targetAgentVersion` with a
     real **SemVer prerelease comparator** (strip a leading `v`, ignore build metadata; naive string `<`
     mis-orders `-beta.10` vs `-beta.2`) → equal ⇒ `applied`, below ⇒ `pending`.
   - `stale` ONLY when last-seen is old AND there is NO `applying` marker (a node mid-self-update
     legitimately goes quiet while it re-execs — require BOTH staleness AND no in-progress marker).

3. **Wire `settings` + chip:** subscribe to `settings` from `useControllerStore` in BOTH `NodeRegistry`
   and `FleetNodeDetailPage` and pass it to `deriveUpdateState`; treat `settings === null` as `off`/no-chip
   (defensive — `refresh()` loads settings best-effort and it may be null on first paint; also satisfies
   the single-ErrorBoundary concern). Render a new column + cell in `NodeRegistry` (modeled on the badge
   cells), colored by severity (applied=green, applying/pending=amber, failed=red, stale=grey,
   off/not-targeted=muted), tooltip with the raw `lastHealth`/reason. Add a matching `<dl>` field on
   `FleetNodeDetailPage`.

4. **Opt-in auto-poll** (decided): a 15–30s interval (place on `FleetPage` or the Shell) that calls the
   existing `refresh()`; **pause when `document.hidden`**; **clear on unmount AND on logout/mode-switch**
   (respect the session gate — never leak requests past logout). Gate it behind a small "Live" toggle
   (default OFF — opt-in per the outline post-flight decision; the operator turns it on).

5. **i18n:** `updateStatus.*` keys for each enum state + the Live toggle, in BOTH `en.ts` and `zh.ts`.

## Verification
- `cd frontend && npm run lint && npm run build` (bijection).
- **Manual smoke:** with a target set, a canary node shows `pending`→`applied` as its reported version
  flips; a node with an `abandoned:` health line shows `failed`; an out-of-rollout node shows
  `not-targeted`; empty target shows no chip; the Live poll updates without manual refresh and **stops on
  logout**.

## Tests produced by this plan
- None automated (no FE runner). `deriveUpdateState` is pure and the prime candidate for a unit test if a
  runner is ever added (out of scope — the deferral stands); verified by `tsc` build + manual smoke.
  Keep `deriveUpdateState` pure + exported so a future runner can cover it cheaply.

## Definition of done
- Per-node chip renders the enum states on Fleet + Node detail, derived correctly (substring `includes`
  for the health markers, real SemVer compare for pending-vs-applied, membership via `inRollout`);
  `failed` is best-effort/transient (noted in the tooltip + closure notes); opt-in poll updates live,
  pauses on tab-hidden, clears on logout.
- Bijection + lint + build green.

## Out of scope

- A rollout history/timeline (rich-dashboard scope, declined). Closure/release (plan-6).
