# plan-8 — Store behavioral core (KEYSTONE, highest risk, LAST)

**Goal:** Collapse the two hand-mirrored 46-method Store implementations into ONE behavioral core over a
thin KV port, so every custody/allocation invariant is authored once and `MemStore` (used in all tests)
finally exercises the telemetry-overlay path that ships. **This is the single genuinely-critical debt
anchor — and the highest-risk move, so it lands LAST.**

**Prerequisites:** plan-2 (`filestore.go` split into io/audit/telemetry) — and all earlier phases green.

## Design (assessment recap)
The 46-method interface (`store.go:497-735`) is the correct chokepoint, but `MemStore` (957) and
`FileStore` (2307) INDEPENDENTLY re-encode every business rule: promote generation-scoping
(`memstore.go:383-420` vs `filestore.go:1174-1296`), the served-vs-staged keystone predicate
("keystone on AND `len(SignatureJSON)>0`," triplicated at `memstore.go:414/750`, `filestore.go:1279/1873`),
monotonic anti-rollback, token rotation, TOTP watermark. A 1559-line `store_compat_test.go` exists ONLY
to police the drift, and telemetry has ALREADY diverged (`filestore.go:691-828` volatile overlay + a
second lock vs `memstore.go:225-244` durable-write, opposite map-aliasing).

## Changes
- New `internal/controller/kv.go` — a ~10-primitive record-CRUD/KV PORT (get/put/list/delete over the
  record types; the only thing a backend must implement).
- New `internal/controller/storecore.go` — the store-agnostic behavioral core: promote scoping, the
  served-vs-staged predicate, monotonic anti-rollback, token rotation, TOTP watermark, the telemetry
  overlay — each implemented ONCE, over the KV port. The interface (`store.go`) stays IDENTICAL externally.
- Reduce `memstore.go` + the split `filestore*` to thin `memkv`/`filekv` adapters.
- **CRITICAL direction (locked decision):** the shared telemetry model PRESERVES FileStore's volatile
  overlay (the deliberate no-fsync-per-heartbeat / no-restart-survival DoS-hardening) — teach the core
  the overlay; do NOT force MemStore's durable-write (a regression two design proposals stated backwards).
- Keep `store_compat_test.go` running; **EXPAND it first** with characterization tests for the overlay
  path, prove the core equivalent, THEN retire its bulk.

## Verify + branch
Full Go suite with `-race` on `internal/controller` + `internal/regression`; the expanded compat test
green throughout the migration, then trimmed; `gofmt -l`. **Owner-gated merge:** a real-host
keystone-rotation + restart + passkey custody smoke (the beta.4–6 fleet-stranding class lives here).
Branch `refactor/plan-8-store-core`.

## Tests produced
- `storecore` unit tests, one per invariant (promote scoping, served-vs-staged, anti-rollback, rotation,
  TOTP, telemetry overlay) — **perpetual** (guard custody/allocation invariants).
- The expanded overlay-path characterization tests fold into the perpetual set; the anti-drift bulk of
  `store_compat_test.go` retires (its purpose is subsumed by "one impl").

## Invariants at risk
- **[5] signed-update custody + served-vs-staged atomic snapshot:** the promote/served predicate +
  monotonic anti-rollback floor MUST live in ONE shared function; a divergence here strands a fleet.
- **[6] zero-knowledge custody + tenant isolation:** the chokepoint semantics preserved EXACTLY.
- **[2] allocation stability** across the generation counters.

## Stop-loss
**plan-8.5** if the extraction reveals a custody/anti-rollback rule that MemStore and FileStore implement
with a GENUINE semantic difference beyond telemetry — STOP, surface it, reconcile with the owner before
collapsing. This is the fleet-stranding-risk phase; there is no "just make it compile."

## Out of scope
Any change to the interface surface (`store.go` externally identical); the FileStore SPOF/HA question
(deferred — architectural, not clean-code); the drift gate for wire DTOs (plan-10).
