# plan-4 — Flip WASM default + soak

**Goal:** Make WASM the local engine; keep the TS compiler present as a one-flag dead fallback for one
release. A working local engine throughout; fully reversible.

**Prerequisites:** plan-3 (WASM alongside + the permanent gate green + registered).

## Design (assessment recap)
The three-way gate has proven WASM == Go golden == TS over the full corpus. Flipping the default is a
single flag change through plan-0's `deployMode.ts`; the permanent WASM-vs-golden gate continues to run,
so the flip is protected. The one accepted open task is WASM live-preview latency (the browser now runs
the whole pipeline per keystroke in local mode) — mitigate with debounce; controller-mode preview stays
server-side.

## Changes
- Default `VITE_YAOG_LOCAL_ENGINE='wasm'` (via `frontend/src/lib/deployMode.ts`); the TS engine remains
  selectable as a fallback for the soak.
- Ensure the local-mode compile call sites (`frontend/src/stores/topologyStore.ts` and any design-preview
  path) route through `wasmEngine` when the flag is default; add debounce on the live-preview path.
- Keep the permanent three-way gate running unchanged (nothing is deleted yet).

## Verify + branch
Full FE suite; an e2e local-mode design→compile→export→deploy-script flow on the WASM engine; the
three-way gate stays green. `npm run build` + vitest + lint. Branch `refactor/plan-4-wasm-default`.

## Tests produced
- An e2e local-mode design flow exercising the WASM engine (`frontend/e2e`) — **subject-scoped** (guards
  the cutover; retires when plan-5 deletes the TS twin and this becomes the only engine, at which point a
  perpetual local-engine e2e supersedes it).

## Invariants at risk
- **[9] two-deployment story:** a WORKING local engine must exist throughout; the TS fallback stays one
  flag away for the whole soak — do NOT remove it here.
- **[1] parity:** the permanent gate continues; a red gate during soak reverts the default immediately.

## Stop-loss
If the WASM engine shows a user-blocking regression during soak (latency, a browser-env incompatibility),
flip the default back to TS (one flag) — reversible by design — and address before re-flipping. Do NOT
proceed to plan-5 (deletion) until the soak is clean.

## Out of scope
Deleting the TS twin / `conformance/` / shedding deps (plan-5); the pure-core splits (plan-5).

## Soak runbook (turnkey — prepped ahead; the OWNER runs the soak)
Two npm scripts stand the WASM engine up as the LOCAL compute engine WITHOUT flipping the shipped default
(TS stays the default until this plan actually merges — the fallback stays one flag away, per invariant [9]):
- `cd frontend && npm run dev:wasm` — builds `web/yaog.wasm`, copies it into `frontend/public/`, and starts
  the Vite dev server with `VITE_YAOG_LOCAL_ENGINE=wasm`. Design a topology locally; every local compute
  (validate / compile / export ZIP / deploy-script preview) then runs through the in-browser Go/WASM engine.
- `cd frontend && npm run preview:wasm` — the same but a production build + `vite preview` (closer to ship).

Clean-soak checklist (the plan-4 flip + the plan-5 deletion WAIT on a clean pass):
1. Validate → same errors/warnings as the TS engine; no console/WASM errors.
2. Compile + export ZIP → bundles open; the permanent three-way gate already proves BYTE-parity, so this is
   the RUNTIME/UX check the headless Node gate cannot make: instantiation, load time, memory on a big topology.
3. Deploy-script preview (sh + ps1) renders.
4. Exercise across the owner's target browsers — **especially Safari/WebKit** (a WebKit-specific WASM quirk
   is exactly what the soak adds over the Node gate).
5. Watch cold-load latency + re-instantiation cost on repeated compiles.
A clean pass on 1–5 = green light to flip the default (this plan) then delete the TS twin (plan-5). Any
user-blocking regression → do NOT flip; report it (the flag reverts in one line).
