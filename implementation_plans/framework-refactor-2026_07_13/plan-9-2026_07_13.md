# plan-9 — Airgap-server retirement (gated)

**Goal:** Delete the anonymous compile/validate/export/deploy-script attack surface (the `//go:build
airgap` server) once WASM is the proven local engine and nothing else needs it — collapsing to ONE
server build and resolving the three-flag two-deployment ambiguity.

**Prerequisites:** plan-4/plan-5 (WASM is the soaked, proven local engine — there must never be a window
without a working local engine). **Owner decision (locked): PROCEEDS — the airgap server is build/test-only,
never hosted.**

## Design (assessment recap)
The two-deployment split is governed by THREE uncoordinated flags across two languages (the Go build tag +
`VITE_LOCAL_ONLY` + `VITE_YAOG_LOCAL_ENGINE`) with no single source validating combinations. The airgap
build serves anonymous compute routes (`POST /api/{validate,compile,export,deploy-script}`) that are
absent from the shipped controller by design — but with WASM as the in-browser local engine, that whole
anonymous surface is redundant. It is currently the `cmd/e2eserver --mode airgap` boot target, so
retiring it is NOT free — that harness must be re-pointed first. (Correction: `internal/dast` boots
`e2eserver --mode controller` only and NEVER touches the airgap server; it just needs `e2eserver` to keep
compiling in controller mode after the airgap files are removed — no re-point.)

## Changes
- **First**, re-point `cmd/e2eserver`'s `--mode airgap` off the airgap server (budgeted here, a real cost,
  not a delete-and-hope); `internal/dast` needs no re-point (it already boots controller mode) beyond
  `e2eserver` continuing to compile.
- **Then** delete `internal/api/handler_airgap.go` / `airgap_routes.go` / `airgap_stubs.go`; collapse the
  two build-tagged boot paths to ONE server build; drop the `backend` local-engine escape hatch in
  `frontend/src/lib/deployMode.ts` (leaving the WASM local engine as the offline design path).
- Update the `boot_default` loud-fail message that currently lists the airgap build as an offline option;
  update `docs/spec/operations/deployment-topology.md` to the single-server + WASM-local story.

## Verify + branch
Full Go suite (now ONE build — no `-tags airgap`); the re-pointed e2e/DAST harnesses green on the new
target; a build-time assertion that the anonymous route class is gone (no `HandleCompile`/`HandleValidate`
anonymous registration linked). Branch `refactor/plan-9-airgap-retire`.

## Tests produced
- The re-pointed e2e/DAST suites (moved, not new) — **perpetual** (their existing lifetime).
- A build/link assertion that the anonymous compute routes are absent — **perpetual** (guards the
  attack-surface deletion).

## Invariants at risk
- **[9] two-deployment story:** sequenced AFTER WASM is the proven local engine so there is NEVER a window
  without one; the WASM static SPA is the offline design path that replaces the airgap server.

## Stop-loss
**plan-9.5** if an e2e/DAST harness cannot be cleanly re-pointed off the airgap server — KEEP the airgap
build and DEFER this phase rather than strand the harness. (Also: if the owner's "never hosted" premise
ever changes, drop this phase entirely and keep the airgap build permanently — invariant 9's edge.)

## Out of scope
Any controller-mode change; the FE polish + drift gate (plan-10).
