# Deploy Scripts

Compilation renders two project-level helpers at the artifact root:

- `deploy-all.sh` for Bash operators.
- `deploy-all.ps1` for PowerShell operators.

The helpers are custody-aware. An **AirGap** compile produces operational SSH helpers. An
**AgentHeld** compile produces fail-closed guidance stubs: they exit without uploading, executing,
cleaning, or uninstalling anything. This split is load-bearing because only `yaog-agent` verifies
the off-host membership signature and durable anti-rollback state used by controller/manual
AgentHeld bundles.

## Artifact layout contract

Every export surface uses the stable node ID as the one per-node directory key:

```text
<artifact-root>/
├── deploy-all.sh
├── deploy-all.ps1
├── <node-id>/
│   ├── install.sh
│   ├── checksums.sha256
│   ├── README.txt
│   ├── wireguard/
│   ├── sysctl/
│   └── ... manifest, plus optional Babel, catalog, and signing files
└── ...
```

There is no node-name fallback and no retired flat `<node>.install.sh` wrapper lookup. The CLI
filesystem exporter, controller stage reader, browser/WASM ZIP writer, and deploy renderer all use
`node.ID` exactly. Validation rejects IDs that are unsafe or ambiguous on supported filesystems;
see [naming.md](./naming.md).

`install.sh` and `checksums.sha256` are required for an AirGap deploy. The checksum file covers the
exact root-executed script and canonical bundle members; a helper skips an incomplete node instead
of uploading a script that cannot pass its preflight. `manifest.json` is metadata and is not the
checksum authority.

The browser/WASM export places the `deploy-all.sh` and `deploy-all.ps1` generated by the **same
compile** at the ZIP root beside the ID-keyed directories they address. Separate UI downloads are
conveniences; the atomic ZIP prevents accidentally pairing bundles with helpers from another
compile.

## AirGap deploy mode

Before extraction, each operational helper rejects non-canonical/traversal paths, duplicate names,
and link or special-file entries. After extraction it rejects links/special files again, because a
recursive SCP must never follow an archive-provided link into the operator host.

For each SSH-configured node, the AirGap helper then:

1. Locates the complete `<node-id>/` directory.
2. Tests SSH connectivity.
3. Creates a unique remote directory from `/tmp/yaog-<node-id>-XXXXXXXX` with `mktemp`.
4. Recursively SCPs the complete directory into `<staging>/bundle`.
5. Runs `sudo bash <staging>/bundle/install.sh`.
6. Removes the remote staging directory after success or failure and records the result.

Nodes without SSH details are skipped. Failures are isolated per node and the helper prints a final
success/skipped/failed summary.

The copied `install.sh` is the AirGap execution boundary. It verifies a configured Ed25519
signature first when signing was enabled, verifies every listed SHA-256 digest, checks root, and
only then mutates the host. See [install-script.md](./install-script.md) and
[../controller/signing.md](../controller/signing.md).

### Clean mode

`--clean` / `-Clean` is used when switching between single-interface and per-peer-interface
layouts. The helper forwards only the fixed `--clean` token to the verified installer. It does not
perform destructive SSH cleanup before authenticating the candidate bundle.

```bash
bash deploy-all.sh --clean artifacts.zip
```

```powershell
.\deploy-all.ps1 -ArtifactsZip artifacts.zip -Clean
```

### Uninstall mode

AirGap `--uninstall` / `-Uninstall` needs no artifact ZIP. It is an explicit operator action: the
helper SSHs into each configured node and runs the topology-specific teardown directly, including
WireGuard, Babel where applicable, mimic/SNAT/dummy state, sysctl configuration, and systemd
reload.

```bash
bash deploy-all.sh --uninstall
```

```powershell
.\deploy-all.ps1 -Uninstall
```

## AgentHeld deployment and manual-node commands

The generated AgentHeld helpers deliberately exit with an error. They do not directly run the
downloaded `install.sh`, including in clean or uninstall mode:

- Managed nodes deploy through controller **stage → off-host sign when keystone is on → promote**;
  their enrolled agents fetch, verify, stage, and apply the promoted bundle.
- Manual nodes use a separately installed `yaog-agent` and a per-node bundle delivered as a
  directory or ZIP. `kit apply` captures an immutable bounded snapshot, verifies bundle integrity,
  off-host membership, node identity, and rollback floors, stages a trusted copy, and executes only
  that copy.

Use the command matching the separately provisioned operator credential:

```bash
# Raw Ed25519 keystone
sudo yaog-agent kit apply \
  --bundle <node-bundle-dir-or-zip> --node-id <node-id> \
  --operator-cred <trusted-public-key.pem> --operator-cred-alg ed25519

# Browser WebAuthn keystone (RP ID is required; origin preserves the enrollment binding)
sudo yaog-agent kit apply \
  --bundle <node-bundle-dir-or-zip> --node-id <node-id> \
  --operator-cred <trusted-public-key.pem> \
  --operator-cred-alg <webauthn-es256|webauthn-eddsa> \
  --operator-rpid <rp-id> --operator-origin <origin>
```

Append `--uninstall` to the same verified command for removal. A legacy fleet that has **never**
enabled a keystone can acknowledge that absence explicitly with
`--dangerously-allow-no-keystone`; the flag is rejected when trust-list files are present or the
node's durable state records prior keystone verification. It is not a keystone-bypass mechanism.

## SSH selection and quoting

An `ssh_alias` uses the operator's SSH configuration. Otherwise the AirGap helpers form
`<ssh_user>@<ssh_host>` and apply the configured port/key. Topology-derived targets, key paths,
node display names, node-ID directory segments, and comment text are escaped for their Bash or
PowerShell context. Remote staging is a fresh node-ID-keyed directory, so concurrent deploys do not
reuse a predictable payload path.

Password authentication is not automated; configure key-based access or an SSH agent before
running an AirGap helper.
