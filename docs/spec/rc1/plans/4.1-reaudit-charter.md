# 4.1 — Post-refactor re-audit charter

> **Status:** authored by plan-20 (Subject 4, milestone 4.1). This document is the **spec** for the
> FINAL pre-rc.1 adversarial security audit. plan-21 (4.2) **executes** it; plan-22 (4.3) consumes its
> verdict as criterion **A2**. It changes no production code and edits no sibling plan (4.2/4.3) — it is
> reconciled to AGREE with them, in particular the `realtunnel` exit-bar amendment (↔ plan-22 / 4.3, R8 / Task 6).

## § Objective

The re-audit is the gate between **"all work done"** (Subjects 1–3 landed) and **"tag rc.1"**. It is run
against the system **as it will actually ship after the refactor** — the in-browser TS compiler
(plan-4/plan-6), the controller-only backend (plan-7, air-gap compute behind `//go:build airgap`), the
Subject-2 phone UX (plans 10–12), and the new TS crypto/ZIP supply chain (`@noble/*`, JSZip) — **not**
against the unchanged round-1 tree. Its discipline is **diff-aware**: round-1's refuted/sound results
are confirmed-by-reference, and the 14 round-1 confirmed findings + B1–B4 are a **regression baseline to
re-verify, not rediscover**. This document authors the spec; plan-21 runs the finders + two-lens verify
+ synthesis and writes `docs/spec/rc1/4.2-verdict.md`; that verdict feeds plan-22 criterion A2 and this
charter's exit bar (which they share, not duplicate).

## § Workflow shape (finder → two-lens verify → synthesize)

Reuse the round-1 shape, stated precisely so plan-21 executes it without re-design:

- **Finders (fan-out, ≤4 concurrent per the MEMORY cap "cap workflows at 5 concurrent agents").** One
  finder per surface-cluster: {auth/session O1+O6}, {custody O4+O2}, {SSRF/egress O3}, {compile-DoS
  oracle O5}, {TS-port new surface N1+N2}, {supply chain N4}. Each emits candidate findings with
  file:line + a concrete exploit sketch. Finders are **diff-aware**: a zero-diff surface with a passing
  baseline re-check is confirmed-by-reference, not re-walked line-by-line. (N3 is a diff-confirmation,
  not a finder.)
- **Two-lens adversarial verify** on each candidate:
  - **Exploit lens** — construct the actual attack path (a request shape, a fixture, a sequence) OR
    refute it (the guard fires *here*, at this line). Neither ⇒ stays UNCERTAIN.
  - **Reachability-skeptic lens** — reachable in the SHIPPED posture, or theoretical? (e.g. S1–S3 are
    anonymous only in a deployment mode plan-7 deletes ⇒ CLOSED-BY-REMOVAL for the deployed surface;
    operator-only disclosure is to a trusted actor.) Scopes blast radius, mirroring round-1 §2a.
- **Synthesize** into: (a) a findings JSON in the round-1 `.security` schema
  (`id`/`title`/`surface`/`severity`/`file`/`line`/`description`, **plus a `disposition` field**) — so
  it diffs against `investigation-findings.json`; (b) a **refuted list** (a refutation is durable, not
  re-litigated next round); (c) the **disposition map** (below); (d) a single **GO / NO-GO verdict** with
  a residual-risk register. plan-21 writes this to `docs/spec/rc1/4.2-verdict.md`; this charter specifies
  the shape.

## § Exit bar (the gate to tag rc.1 — the fix this plan applies)

The audit's **GO** verdict requires ALL of:

1. **Zero open HIGH** anywhere in the inventory (O1–O7 + N1–N4).
2. **Zero unmitigated MEDIUM on the NEW surfaces N1–N4** — a residual MEDIUM on a new surface needs an
   explicit **owner sign-off** (the residual path below).
3. **Every baseline finding (14 + B1–B4) dispositioned AND re-verified** with its executable re-check —
   no silent regression of a beta.8/plan-8 fix (diff-aware static **AND** DAST).
4. **Every Subject-2 endpoint/flow audited** (N3 confirmation: no new secret-storage / CSRF / XSS path).
5. **The N1 invariant holds** — no browser value reaches a trusted sink without server re-validation;
   the TS validator is not security-weaker than Go on the reject path (confirmed via plan-5's byte-equal
   verdict-code-sets, NOT a new harness assertion).
6. **N2 confirmed** — the anonymous compute/keygen routes are route-absent (404) on the shipped
   controller; the static-SPA local deployment has no backend listener.
7. **N4 supply chain green** — the JS/npm SCA over `@noble`/JSZip is wired + clean (or allowlisted with
   rationale); `govulncheck ./...` clean.
8. **Verdict = GO.**
9. **Whole-program build/test gates green AND `plan-5` (1.5) conformance + `plan-18` (3.6) `realtunnel`
   green-and-required.**

> **THE FIX (4.1 exit-bar amendment).** The prior stub's bar read "build/test/conformance/E2E gates
> green" and **omitted `realtunnel`**. Per the locked owner directive (2026-06-18) making plan-18 / 3.6

> plan-18 owner_flag; if it is not yet wired as a distinct required check at audit time, that is itself a
> blocker the exit bar surfaces.)*

## § Owner sign-off path for accepted residuals

This mirrors plan-22 / 4.3's exception model (its owner sign-off block / single owed-smoke ledger) — the **single** exception ledger, no second
conflicting one. Any finding the audit classifies at the **blocker bar** (fleet-availability,
fleet-trust, or controller-mode security/correctness) either:

- **(a)** lands a fix → forces a **beta.9** and a **re-run** of the audit; or
- **(b)** gets a **written, signed owner-accepted-risk exception** with rationale recorded in
  `STATUS.md` + the release notes.

A **residual MEDIUM on N1–N4**, or the **`realtunnel` netns canary** if the owner judges it too
CI-fragile to hard-block on, are the ONLY categories that may cross the tag un-completed — and only with
an explicit owner signature. This is the SAME signature surface 4.3 owns (the charter does not open a
second exception ledger).
